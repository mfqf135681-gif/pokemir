## 阶段 1：需求理解与复述

**用户需求**：项目跑通后要为每个对手建立画像（VPIP / PFR / 3-bet 等 15+ 指标）。**前提是能跨手稳定识别同一个对手**。当前实现只用座位号，跨手等于完全失忆——必须升级。

**业务背景**：
- 当前 `player_stats_cache.player_name` 是 UNIQUE 主键（行业惯例：以 player_name 聚合统计）
- 当前写入数据用 `f"Player_{seat_idx}"`（座位号每手洗一次），所有"画像"统计都会被搅在一起
- 后果：哪怕跑了 10000 手，统计也只是一堆"Player_0/1/2/3/4/5"的 6 个混合体——完全无意义

**核心目标**：决定一个**用户能落地、我能实现、识别准确率够用、不踩 R-1 ToS 红线**的方案。

- DOM 注入 / JS hook / page modification
- 网络抓包（mitmproxy 等）
- 进程内存读取

### 2.7 ⭐ Hand-Start 黄金窗口（基于 2.5 信息推导）

在 `StateTracker.check_hero_cards == True` 的瞬间（hero 卡刚出现 → 新一手开始）：
- 没有玩家行动过 → ID 没被"跟注/加注"遮挡
- 没有玩家弃牌 → 头像未变灰
- 所有 ID + 昵称均在裸态可见

这是抓 player_id 的**唯一可靠时机**。每手抓 1 次 → 在该手内所有 action_events 复用这个 ID 映射。

## 阶段 3：方案设计（5 个，基于 v2 新信息重排）

### 方案 A — 仅 OCR 昵称

**做啥**：每座位 +1 个 `nickname_area`，每手 OCR 1 次。`player_name` = OCR 字串。

**触发红线**：R-7（ROI 结构）、R-8（识别配置：加中文 OCR 模型）

**优点**：实现简单（1-2 天）；ROI 增量小

**缺点**（v2 重估）：
- 昵称是图片 + 中文 + 可能含 emoji → OCR 错率高（行业典型 5-15%）
- 昵称可改：同一人前一会话叫"张三"后一会话叫"张3" → 同人两条记录
- 重名：两个不同人都叫"张三" → 撞名混合统计

### 方案 B — 仅头像 hash

**v2 评估**：**几乎不可用**

**理由**：
- 默认头像撞库本就严重
- **加上 2.5#3 弃牌状态扰动**（变灰 + 叠"弃牌"字）→ 同人 active vs folded 头像 hash 完全不同 → 同人识别成两人
- 此外还有计时圈、talking dot 等动态元素扰动
- 即使能解决状态扰动（需要复杂的 active-only 抓取时机），撞库率仍高

**结论**：**Drop from consideration**。可作为 C/D 的弱兜底信号，但不能单独成方案。

### 方案 C — OCR 昵称 + 头像 hash 组合（原行业标准）

**v2 评估**：因为 B 几乎不可用，C 退化成"A + 一个不靠谱的兜底"——不再是行业标准等价物。**不推荐**。

### 方案 D-revised — Hand-Start 窗口 OCR ID ⭐ 新主推

**做啥**：
1. 给 `SeatROI` 加 `id_area: ROIRegion` 字段
2. 用户 ROI 配置时每座位框 1 个 ID 区
3. pipeline 在 `check_hero_cards == True` 的瞬间（hand-start，详见 2.7）扫一遍所有座位的 `id_area` 做 OCR
4. OCR 出的 ID 数字串作为该手的 `player_id` → 写进 `action_events.player_name`
5. 这手内后续所有动作都关联到这个 ID

**为何选这个时机**：阶段 2.7 已解释，hand-start 的瞬间是 ID 唯一未遮挡的窗口。

**触发红线**：
- R-7（ROI 结构）：触发
- R-8（识别配置）：触发；但数字 OCR 现有 easyocr 英文模型零配置即可
- R-5/R-6：**不触发**（player_name 字段已存在，只是把内容从座位号改成 ID 数字）

**优点**：
- ✅ ID = 平台稳定全局唯一标识 → 跨手 / 跨会话识别 100% 准
- ✅ 数字 OCR 准确率 99%+（远高于中文昵称）
- ✅ 不撞名 / 不撞库
- ✅ 头像状态扰动 / nickname 改名 都不影响
- ✅ ROI 增量小（每座位 +1 框）
- ✅ DB schema 零改动（Phase 1 简单的"player_name 字段存的是 ID 数字"即可）
- ✅ DEV 工作量 1-1.5 天

**缺点**：
- ❌ 玩家在该手内行动后 ID 被遮挡 → 当前手只能在 hand-start 抓 1 次，错过就没机会；但因为每手新开始时都会有 1 次窗口，错过这手下手能补
- ❌ 如果 hand-start 时某座位刚 post SB/BB，可能 ID 已被 "Posting SB" 字样部分遮挡（**待用户确认**）
- ❌ player_name 在 DB 里全是数字，人眼看 SQL 不直观（缺 display_name）

### 方案 D++ — D-revised + 同时 OCR 昵称用于显示

**做啥**：在 D-revised 基础上，hand-start 同瞬间也 OCR `nickname_area`（每座位 +2 ROI 框）。
- `action_events.player_name` 仍存 ID 数字（聚合 key）
- 另开一个轻量内存映射 `id → last_seen_nickname`（不存 DB，每会话重建）
- 跑 stats 报表时用映射 join 显示昵称

**触发红线**：R-7 / R-8 同 D-revised；不触发 R-5/R-6（不改 schema）

**额外工作量**：+0.5 天

**ROI 增量**：每座位 +2 框（id_area + nickname_area）

### 方案 E — 不实现（维持现状）

**含义**：跨手统计死路；Phase B 对手画像功能不可行；A 路径名存实亡。

**结论**：**不是 viable 方案**——本讨论开了就不能选 E（除非用户决定项目放弃画像）。

### 推荐：**D-revised 起步，按你 ROI 配置耐受度决定是否一步到位 D++**

**理由**：
1. 用户 2.5 信息直接证明"ID 是稳定全局标识，OCR 可行"——这是 best player_id 信号
2. 数字 OCR 准确率远高于中文昵称 OCR
3. Hand-start 窗口（2.7）解决了"被遮挡"问题
4. ROI 增量小（+1 框/座位 = 6-max 桌 +6 框）
5. Phase B 对手画像统计可以基于 ID 稳定运行；display 名字可后续叠加（D++）

- ❌ 不做（留给后续阶段）：DB schema 加 `display_name` 字段（避免触发 R-5/R-6）、nickname OCR（如 Q1 选 D-revised）、头像 hash 兜底、replay 校正 UI
