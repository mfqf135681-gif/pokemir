🔧 **前置条件**：两机能 SSH 互通；Win 启用内置 OpenSSH server。**实际可行性取决于网络拓扑**（待用户答）。

---

### ❓ 决策 1：代码从 Linux 流到 Windows 走哪条路？

🎯 **选项**：
- **A：纯 git push/pull**（原计划）—— 零配置；但每改一行就 commit 会让 git log 堆"wip"碎片；忘 push 时 Win 跑老代码无察觉
- **B：git + rsync 双轨**（推荐）—— 30 分钟一次性配置；之后"改完一存 → 1 秒同步"；git history 保持干净
- **C：换 WSL2 单机**—— 一天迁移；之后再无同步问题；但 mss/ctypes 仍只能在 Win host 跑（不在 WSL 里）

💡 **建议 B**：用 30 分钟一次性配置换"改完即可同步"的体感。但需要你的网络拓扑前置回答。

---

### ❓ 决策 2：要不要现在建"截图 fixture 库"？

🎯 **选项**：
- **A：现在建，作为下一个 DEV 任务**（推荐）—— Win 端录 20-30 张真实截图 + 标注预期；之后每次改识别代码 `pytest` 量化告诉你识别率涨/跌；前期投入半天到一天
- **B：先不做，先修 P0 bug**—— 省半天；但每次修识别代码只能肉眼判断改对没有；同一 bug 容易反复出现
- **C：跳过识别路径，转 hand history 离线分析**—— 放弃实时识别方向；fixture 不再需要；但项目目标变

💡 **建议 A**：刚激活完整治理体系，下一个识别任务进 DEV 就会撞上 §5.2"测试必须全通过"——没 fixture 那条规则形同虚设。fixture 库是激活完整版后的基础设施补课。

---

### ❓ 决策 3：先装哪几个 MCP？

🎯 **选项**：
- **A：只装 filesystem + postgres**（推荐最小集）—— 解决"我看不见 Win 文件"和"我看不见 DB 状态"两个核心痛点
- **B：A + fetch**—— A 能力 + 我能直接拉网页；但 hf-mirror 已解决主要网络问题，fetch 收益低
- **C：A + sequential-thinking**—— 加显式推理链；当前阶段意义不大

💡 **建议 A**：filesystem + postgres 各自解决一个明确痛点。MCP 池太大反而干扰选择。

#### 脚本设计要点

| 维度 | 设计 |
|:---|:---|
| 装的 MCP 集合 | github + context7 + semgrep + filesystem + postgres |
| filesystem 路径 | 默认 `$(pwd)`，可 env var `POKEMIR_FS_PATH` override |
| postgres DSN | 走 `read -s` 交互输入 或 `POKEMIR_PG_DSN` env var；不写明文密码到脚本 |
| context7 / semgrep / github | npx -y，零配置；context7 = user scope 跨项目通用 |
| scope 策略 | filesystem / postgres = `local`（项目特定）；context7 / semgrep / github = `user`（跨项目通用） |
| 幂等性 | 跑前 `claude mcp list` 检查；已装的跳过；可重复执行 |
| 错误处理 | 任一步失败就报错退出，不静默 |

#### 不装项的明确

cloudbase / chrome-devtools / Google Drive **不在本脚本范围**——pokemir 不需要。
