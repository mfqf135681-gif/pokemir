## 阶段 3：方案设计(3 个子决定)

### 3.1 R-3 措辞调整(主决定)

**当前**:"本地手牌数据不外传 / 向非 localhost 发送 → 触发"

**候选改法**:

| 选项 | 措辞 | 优 | 缺 |
|:---|:---|:---|:---|
| **A.** | "数据仅限**用户自控基础设施**(localhost / 用户拥有的 VPS / 私有租户)" | 清晰原则;允许 VPS 与 SSH 隧道;**仍排除公有 SaaS**(Supabase 等运维可读) | 边界判定需用户自我裁定 |
| B. | "白名单制:`.env` 显式声明的远程目标可送" | 极简清晰 | 用户每次新增目标都要更新 |
| C. | 取消 R-3 整条 | 完全自由 | 失去"敏感数据保护"防线 |

**推荐 A**。**理由**:对齐用户意图("自控"是核心),且自然排除掉 Supabase 这类"第三方运维可读"的方案。

### 3.2 私有云具体方案选择

| 方案 | 实施 | 月成本 | 用户控制 | 推荐 |
|:---|:---|:---:|:---:|:---:|
| **A. 用户 Linux VPS (puzz-virvm)** | 已有 docker-compose PG,改远程访问 | ¥0 增量 | ✅ | ⭐⭐⭐⭐⭐ |
| B. Supabase 私有 project | 注册 → 创建 project → 获 DSN | ¥0 (free tier) - ¥xx | ⚠️ Supabase 运维可读 | ⭐⭐ |
| C. Neon serverless | 类似 B | ¥0 - ¥xx | ⚠️ 同上 | ⭐⭐ |
| D. 新 DigitalOcean Droplet ($5/月) + 自装 PG | 新机+装机+安全配置 | ¥35-40/月 | ✅ | ⭐⭐⭐ |

**推荐 A**。**理由**:
1. 已支付,无增量成本
2. 100% 用户自控,符合 R-3 新措辞 A
3. PG 已 docker-compose 在跑(Linux dev 用)→ 配置改动小
4. 我直接 mcp__postgres__query 能查(调试便捷)

### 3.3 网络架构(Win → VPS PG 怎么连)

| 方案 | 安全 | 易用 | 推荐 |
|:---|:---:|:---:|:---:|
| **α. Tailscale VPN(零配置 mesh VPN)** | ✅ 端到端加密 / 私有 IP / 零公网暴露 | ✅ 装 Tailscale 客户端,登 Google/GitHub OAuth → 自动入网 | ⭐⭐⭐⭐⭐ |
| β. SSH 隧道 | ✅ 加密 | ⚠️ Win 要持续跑 ssh -L,断了 pipeline 也断 | ⭐⭐⭐ |
| γ. PG 直连 + TLS + IP 白名单 | ⚠️ 公网暴露 + IP 限制 | ⚠️ Win IP 动态变化时麻烦 | ⭐⭐ |
| δ. WireGuard 自建 VPN | ✅ | ❌ 配置复杂 | ⭐ |

**推荐 α Tailscale**。**理由**:
1. **零摩擦**:Tailscale 客户端 Win + Linux 各装一次,OAuth 登一次,完成
2. **零公网暴露**:不打开 VPS 防火墙 5432 端口
3. **自动重连**:网络中断后自动恢复(比 SSH 隧道稳)
4. **免费**:个人最多 100 设备
5. **R-3 精神保持**:数据从 Win → 加密 mesh → 用户 VPS,**全程用户自控**

### 总推荐:R-3-A + 方案-A + 网络-α (Tailscale)

### 实施步骤(本 REQ confirmed 后切 DEV 做)

1. **Linux VPS 上**:
   - 装 Tailscale: `curl -fsSL https://tailscale.com/install.sh | sh` + `sudo tailscale up`
   - PG docker-compose 改 listen address (从 `127.0.0.1:5432` → `100.x.x.x:5432`, Tailscale 私网 IP)
   - 验证可达
2. **Win 上**:
   - 装 Tailscale Win 客户端
   - 用同账号 OAuth 登,自动加入 mesh
   - `ping vps-tailscale-ip` 测连通
   - 修改 `.env` 中 DSN: `postgresql://poker_user:pwd@<vps-tailscale-ip>:5432/poker_assistant?sslmode=require`
3. **代码**:
   - 修改 `.agents/project-constraints.md` R-3 措辞
   - 测试 pipeline 远程写入正常
   - 验证 pytest test_storage.py 用新 DSN 通过

### scope 边界

- ❌ 不更换 VPS / PG version
- ❌ 不引入 PgBouncer / 连接池优化(等 path B 真用上再说)
- ❌ 不实现数据备份策略(独立 sub-REQ)
- ✅ 包含:R-3 改 + Tailscale 装 + .env 改

### 潜在风险

| 风险 | 严重度 | 缓解 |
|:---|:---:|:---|
| VPS 网络抖动 / 宕机 → pipeline 写入失败 | 🟡 中 | 已有 `_db_enabled` 探测机制 → fallback no-db 不崩 |
| 网络延迟(Win → VPS) 让 100ms tick 失效 | 🟢 低 | pipeline 在 try/except 中,慢但不崩;批写可作为 path B 优化 |
| Tailscale 服务商账户被攻击 | 🟢 低 | 用 2FA;Tailscale 本身不存数据,只做控制平面 |
| 用户失去 VPS 访问 | 🟢 低 | VPS 上 dump 可定期 git 备份(异步任务) |
