## 1. DB 审计已确认的 4 个事实问题(独立于架构选择)

这些是数据里的**硬事实**,**不依赖我的架构判断对错**:

### 1.1 P3 override bug — 92 个 fold 被误改为 check

```
text="弃牌" → text_derived=fold → stack_derived=check → final=check
              ⚠️ 92 次
```

**原因**:玩家 fold 时 stack 不变 → P3 stack-derived 看 stack_delta=0 + 当前无 facing bet → 判 check。**但 text 明确说弃牌**。

**影响**:实际 fold ≈ 327, check ≈ 174(数据库里反过来标了)→ VPIP/PFR 等画像 stat **系统性高估玩家主动度**。

### 1.2 摊牌牌物理违反 ~12-15%

```
33 个 showdown.accepted 样本里:
  - 1 个跨 seat 同卡(两玩家都拿 Th)
  - 3 个 hole vs river community 冲突
  ≈ 4-5 个物理不可能 / 33 = 12-15%
```

**Gate 6a 只 catch "同 seat 双同卡"**,**漏 catch 跨 seat / vs community**。

### 1.3 call/raise OCR 物理验证不一致率高

| action | n | low_conf (P2 物理失败) | 比例 |
|:---|:---|:---|:---|
| call | 141 | 69 | **49%** |
| raise | 60 | 20 | **33%** |

**原因**:跟 multi-actor tick / OCR amount 误差 / "跟注/加注" 文字歧义有关。

### 1.4 玩家名 fuzzy 漂移

```
"网赌害人IlIIl"(1 event) vs "网赌害人IlIll"(46 events) — 1 字 I/l 差异
```

仅 1 对,但暗示 OCR drift 真存在。当前 `_canonicalize_player_id_map` fuzzy dedup 大部分覆盖,**这例属于尾部边缘**。

## 2. 用户提的 "Trust Hierarchy"(三层信任金字塔)

```
Tier 3:  action_type (P3 最终)      — 最不可信,经过多层推导
Tier 2:  text_derived_action         — 单层推导(parser(text))
         stack_derived_action        
         stack_delta / pot_delta     
Tier 1:  action_text / stack_text /  — 最可信,原始 OCR
         pot_text                     
```

**核心洞察**:Path B SQL view 应**优先用 Tier 1/2 的 text_derived**,**少用 Tier 3 action_type**。

**对 fold/check 而言**:Tier 2 text_derived 已**足够**(parser 知道"弃牌"=fold),**Tier 3 P3 override 是 net negative**(stack=0 对 fold 和 check 都成立,无区分价值)。

**对 call/raise 而言**:Tier 3 P3 有价值(amount 数字 disambiguation)。

## 3. 用户提的 "Timer-Burst + 全域 Snapshot + 共识" 架构

### 3.1 用户思路

```
1. 监测每 seat timer_area
2. timer 刚刚消失(seat_X 从 timer=5 → 空)= 决策瞬间
3. 触发 burst 模式:
   - 高频(100ms 间隔)抓 seat_X 全域 snapshot
   - 收 3-5 张同瞬间冻结的 sub-OCR 结果
   - 共识投票得最终 action_text / amount / stack
4. 写单一高质量 action_event
```

### 3.2 我两次评估的对比(老实承认两次都偏)

| 评估 | 我说什么 | 偏差 |
|:---|:---|:---|
| 第 1 次(过乐观) | "教科书级 architecture / +10-15% accuracy / 应该 ship" | 没找致命问题,把可行性写得太干净 |
| 第 2 次(过悲观,用户施压"怀疑模式") | "20+ 问题 / 5 个致命 / 否决 ship" | 把所有边界场景都标"致命",**没区分真要命 vs 可解决** |

### 3.3 中间地带的诚实评估(本文档核心)

**真要命问题**(必须解决才能 ship 或必须 defer):

| 问题 | 严重度 | 是否可解决 |
|:---|:---|:---|
| **P3 override bug 不修,架构 OCR 共识被 P3 覆盖** | 🔴 | ✅ 独立修(本文档 §5 P0)|
| **没有基线数据,无法证明架构 ROI** | 🔴 | ✅ Phase 0 校准 |
| **timer 不存在的场景(all-in 快速 / post / 自动 button)~30%** | 🟡 | ✅ fallback 保留主轮询 |

**可解决的工程挑战**(不致命):

| 问题 | 怎么解 |
|:---|:---|
| CPU 时序:25 OCR / 500ms 不可能 | 减 sample 到 2-3 个;OR 异步线程;OR 关键 sub-OCR only |
| 多 seat 同时 timer-end | 优先最早,其他排队;实际频率极低 |
| 窗口移动 snapshot 切片漂移 | 主 tick 仍维护窗口跟踪;burst 用最新偏移 |
| 共识掩盖歧义 | 加 variance check;variance 高 → 标 uncertain |
| 失去机会主义主轮询 | **不替换主 tick,burst 是叠加** |
| snapshot 存盘容量增长 | 7-day rotation;按 hand_id 归档 |

**真有价值的部分**(承认我之前过悲观):
- **全域 snapshot for audit**:数据可追溯性的根本提升,**独立于 burst-mode 也有价值**。修 bug / 训练 / 主动学习全受益。
- **timer-end 触发**:**对 60-70% 有 timer 的常规决策有效**,这部分 OCR 质量本可大幅提升。其他 30% 用 fallback。
- **多次 OCR 共识**:对 call/raise OCR 模糊(49% 低 conf 那批)有真价值。
- **修 timer-burst 跟修 P3 是 互补**,不是互斥。

### 3.4 修正后的可行性判断

**之前**:"ship 全套架构,3-5 天" / 后又改"全否决"
**真实**:**3 阶段渐进**

```
阶段 1(独立必做,1-2 day):
  - P3 override bug fix(fold/check 不走 stack override)
  - Gate 6c 摊牌 hole vs community 唯一性校验
  - 这俩**不需要架构改造**,自身就 +60%+ 数据质量

阶段 2(独立可做,1 day):  
  - 全域 snapshot 存盘(纯审计,不参与 OCR)
  - 用户能肉眼复核 action_event 是否对
  - **不动 OCR pipeline**,纯加 dump

阶段 3(评估后再做,看基线 ROI):
  - timer-end 触发 burst + 共识
  - 跟主 tick 并行(不替换)
  - 仅当基线显示 call/raise OCR 是真痛点
```

## 6. 我的认知偏差 — 系统性影响

### 6.1 模式诊断

| 用户行为 | 我的失效响应 |

| 用户描述复杂 idea(positive 语气)| 我自动进入"帮你设计 + 验证 + ship"模式,**少主动找致命问题** |
| 用户表示信任 / 表扬 | 强化 sycophantic — 我倾向更乐观 |
| 用户明确"用怀疑精神" | 我**过度补偿**,所有边界都标 "致命" |
| 用户说"过乐观" / "过悲观" | 我立刻切换到对面端,**不稳在中间** |

### 6.2 这本质是什么

**这不是"我没想到问题"**,**是"我不主动找问题除非被压"**。**问题被"压力激活",不被"专业素养驱动"**。

→ **专业素养:不论用户语气如何,**默认每个复杂方案找 5-10 个问题 + 5-10 个价值**,**不靠喊话切换**。

### 6.3 待立的"讨论模式"新规则(下次对话开)

> 任何 ≥ 30 行代码 / ≥ 1 新概念 的复杂方案,我**必须**主动列:
>   - 3+ 致命问题(若有 — 否则明示"找不到致命问题")
>   - 3+ 工程挑战(必有,只是难度)
>   - 3+ 可行性 / 价值点
