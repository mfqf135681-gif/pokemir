## 阶段 1：需求理解与复述

**用户需求**:stage B 已 accept,但发现纯依赖 `action_text` OCR 有几个数据完整性问题:
- raise vs bet 区分模糊(WePoker UI 偶尔不显示"加注"字)
- 中文 OCR 偶有误识别
- 单一信号无校验,错答案污染未来 path B(VPIP/PFR)统计

**核心目标**:用**多路冗余信号 + 物理/规则约束**,把 action_events 数据质量从"~85% 单一文字 OCR"提升到"~99% 多路一致 + 标记低 confidence"。

**业务背景**:
- 已有 DB schema:`hands` / `action_events` 完整;`replay_corrections` 表早就准备好(action_type/amount/cards 都可修)
- 已知问题:用户观察确认 seat 真加注但 0 落库;数据样本里 100 actions / 0 raise(post-fix 后修到 1 raise,但 sample 太小)
- 已有信号:**action_text** (已用,易错) + **stack** (已 OCR,未用 delta) + **pot** (已 OCR,有 latest_pot_bb 字段)
- **关键架构原则**(用户讨论 confirm):**不要找单一裁判;多层防御金字塔;最后标 review**

**约束**:
- ✅ R-3 不破坏 — 仅本地计算多信号,无外传
- ✅ R-7 — schema 改加列必经 REQ + change-log
- ⚠️ 已有 stage B 数据(post-fix 25+ events)语义不变,但新字段值会 NULL
- ⚠️ 用户体力:渐进式 4 阶段 P1-P4,允许多次会话

**模糊点(阶段 5 待澄清)**:
- P1-P4 一次性做 vs 分阶段
- 旧 action_events 数据 backfill 与否
- confidence threshold 数值(0.5? 0.7?)
- P3 严格模式 — stack_delta 与 text 冲突时强 override 还是仅打 confidence

## 阶段 3：方案设计

### 3.1 4 层金字塔(架构原则)

```
┌──────────────────────────────────────────────┐
│ Layer 1: 物理方程验证                          │
│   pot_delta == sum(stack_deltas) ?           │
│   → 自洽时 confidence=1.0,无需投票           │
├──────────────────────────────────────────────┤
│ Layer 2: 扑克规则过滤                          │
│   - facing bet 不可能 check                   │
│   - call ≥ to_call;raise ≥ 2 × prev_bet     │
│   → 违规信号自动淘汰                          │
├──────────────────────────────────────────────┤
│ Layer 3: 数字优先级 cascade                    │
│   stack_delta > pot_delta > text             │
│   → stack OCR allowlist 简单更稳定            │
├──────────────────────────────────────────────┤
│ Layer 4: 标记 review                          │
│   confidence < 阈值 → replay_corrections      │
│   → 不强裁,等离线人工修                       │
└──────────────────────────────────────────────┘
```

### 3.2 推断 action_type 的规则(基于 stack_delta + 街道状态)

| stack_delta | current_to_call | is_first_bet_this_street | 推断 action |
|:---:|:---:|:---:|:---|
| 0 | 0 | 任意 | **check** 或 **fold**(用 fold_area 区分)|
| 0 | > 0 | 任意 | **fold**(被 bet 时不出筹码 = fold)|
| > 0 = to_call | > 0 | 否 | **call** |
| > 0,无 prev bet | 0 | 是 | **bet** |
| > 0 > to_call | > 0 | 否 | **raise**(amount=stack_delta)|
| = full stack | 任意 | 任意 | **all_in** |

**核心洞察**:bet vs raise **零关键字依赖**,只看"本街道是否首次出筹码"。WePoker UI 文字怎么变都无所谓。

### 3.3 分阶段实施(P1-P4)

#### P1:信号采集 + 持久化基建(~30 行)

| 改动 | 文件 |
|:---|:---|
| `action_events` 加 `confidence_score FLOAT DEFAULT 1.0` | `contracts/models.sql` + `storage/models.py` |
| ALTER TABLE migration(VPS PG)| 直接 SQL,nullable |
| `StateTracker` 加 `_prev_stack: dict[int, float]`(per-seat 上次 OCR)| `pipeline/detector.py` |
| `StateTracker` 加 `_street_first_bettor: dict[Street, bool]` | 同上 |
| orchestrator `_process_seat_actions` 把所有原始证据写 `raw_data` JSONB:`{action_text, stack_before, stack_after, stack_delta, pot_before, pot_after, pot_delta, current_to_call, is_first_bet, parsed_action_type}` | `pipeline/orchestrator.py` |

**P1 验证标准**:跑 pipeline,DB 中新 action_events 的 raw_data 含 5+ 个键。不破坏现 action_type 逻辑。

#### P2:Layer 1 物理方程 + confidence 评分(~20 行)

| 改动 | 内容 |
|:---|:---|
| 在 `_process_seat_actions` 写 event 前,加方程检查 | `abs(pot_delta - stack_delta) <= tolerance`(tolerance=2 chips 应对 OCR 抖动)|
| 自洽 → `confidence_score = 1.0`;不自洽 → 0.7;completely conflicting → 0.3 | 写入 DB |
| 加 SQL 索引 `idx_action_events_confidence` | 方便 Layer 4 query |

**P2 验证**:数据库中开始出现非 1.0 的 confidence 值;SQL 抽 `SELECT * WHERE confidence_score < 0.7` 看分布。

#### P3:Layer 2 + Layer 3 规则推断(~30 行)

| 改动 | 内容 |
|:---|:---|
| 新加 `event_normalizer.infer_action_from_delta(stack_delta, to_call, is_first_bet)` | `events/normalizer.py` 或 `pipeline/detector.py` |
| 在 orchestrator 主路径:**先**调 infer 函数,**得到 stack-derived action_type**;**再**调 parser 拿 text-derived;两者比对 | `pipeline/orchestrator.py` |
| 优先级:stack-derived > text-derived(数字优先);冲突时 confidence=0.5 + raw_data 写双方意见 | 同上 |
| 在 raw_data 加 `stack_derived_action / text_derived_action / chosen_action / disagreement_reason` | 完整审计追溯 |

**P3 验证**:观察 raise vs bet 自动分类;raise 检出率应 ≥ 95%(假设 stack OCR 准)。

#### P4:Layer 4 review 工具 + 离线修正(~20 行)

| 改动 | 内容 |
|:---|:---|
| 新加 `tools/replay_review.py` CLI | 拉低 confidence 事件 + raw_data 显示;接受 y/n/edit 输入;写 `replay_corrections` 表 |
| SQL view `low_confidence_events` | 方便快速查询 |
| (可选)Streamlit UI 显示截图证据 + 修正按钮 | future polish |

**P4 验证**:能 review 10 条低 confidence event 并写回 replay_corrections。

### 3.4 推断与现状的对比

| 场景 | 现状(纯 text)| 4 层金字塔后 |
|:---|:---|:---|
| WePoker 加注 + OCR 读"加注 50" | parser → raise(对) | Layer 1 自洽 → raise(对,confidence=1.0) |
| WePoker 加注 + OCR 读"下注 50" | parser → bet(**错**) | Layer 2 规则:本街非首次出筹码 → 强推 raise(**对**,confidence=0.7) |
| WePoker 加注 + OCR 读 "" (空) | 无 event(**丢**) | stack_delta > 0 → infer raise(**得救**,confidence=0.5) |
| 多玩家同 tick 行动 | latest pot 噪声 | stack_delta per-seat 不互扰(对) |
| OCR stack 偶发错读 | 不影响(因不用)| Layer 1 不自洽 → confidence 降 → 标 review |

→ **数据质量提升 + 鲁棒性 + 异常可见性**三赢。

## 阶段 4：对比与推荐

### 总推荐:**渐进式 P1 → P2 → P3 → P4,每阶段独立 commit + 验证**

**理由**:
1. 每阶段产物可独立 SQL 验证
2. 出问题易回滚
3. 4 个阶段并不强相关(P1 是基础;P2-P4 可独立)
4. Stage B 已 accept,本 REQ 是衔接 path B 的关键质量补丁,**不阻塞继续观战数据**

### scope 边界

- ❌ 不动 recognition/* 模型层
- ❌ 不动 capture/* ROI 采集
- ❌ 不实现 Streamlit UI(P4 仅 CLI;UI 留独立 sub-REQ)
- ❌ 不 backfill 旧 action_events(旧数据 confidence=NULL = 未评估,path B 统计可忽略;backfill 是高风险脚本,不在本 scope)
- ✅ 含:schema 加 1 列 + 5 个 raw_data 键 + 4 阶段渐进代码 + CLI review 工具

### 实施 commit 序列

```
[REQ confirmed]
  ↓
[P1 commit: schema + raw_data 写入]    ← 不改 action_type 逻辑,只采证据
  ↓
[P2 commit: Layer 1 自洽检查 + confidence 评分]
  ↓
[P3 commit: stack-derived action + text-derived 对比 + 优先级]    ← 核心质量提升
  ↓
[P4 commit: review CLI + replay_corrections 写回]
  ↓
[REQ accepted: 多信号交叉验证全栈]
```

### 潜在风险

| 风险 | 严重度 | 缓解 |
|:---|:---:|:---|
| stack OCR 偶发错读 → P3 错改 action_type | 🟡 中 | Layer 1 自洽检查兜底;不自洽时 fallback text + 标 confidence 低 |
| confidence 阈值选错 → review 队列爆炸或漏检 | 🟢 低 | 从 0.5 开始,实测后调整 |
| schema migration 影响 pipeline 启动 | 🟢 低 | ALTER TABLE ADD COLUMN IF NOT EXISTS + NULL 默认,向后兼容 |
| P3 改动语义影响 stage B 已 accept 数据 | 🟢 低 | 新逻辑只影响**未来** events,旧的 confidence=NULL = 未评估,path B 自然过滤 |
