## 阶段 1：需求理解与复述

**用户需求**：识别 baseline 当前 48%（15/31 pass），剩 13 fail（全部同色花色互窜）+ 3 skip（OCR 字体盲点）。继续手搓启发式 ROI 阈值 trade-off 大、风险高、调出来还不一定鲁棒。问能否换更高效模型一步到位（或大幅）提升。

**业务背景**：
- 当前栈：EasyOCR（识别 rank 数字）+ 手搓 HSV 颜色判定（识别 red/black）+ Stage 1 多 crop fallback
- Stage 2（同色花色 d↔h / s↔c 区分）实测需要更高级的形状特征——简单像素分布扛不动
- 用户暗态卡（[[wepoker-card-brightness-variance]]）让纯启发式更脆弱
- 任何模型必须**满足 R-1 / R-3 / [[image-only-compliance-constraint]]**：本地推理，不联网，不抓 DOM

**核心目标**：决定**是否换识别栈**，以及换的话用哪个方案。

**约束**（v2 完整版，含用户新提供算力信息）：
- ✅ 必须本地推理（无云 API）—— R-3 手牌数据不外传
- ✅ 必须可纯靠屏幕截图工作 —— [[image-only-compliance-constraint]]
- ⚠️ **Linux VPS 算力有限**——重模型在 VPS 跑可能 OOM / 慢到不可用 → **不能在 VPS 上做实测验证**
- ✅ **Win 测试机算力较强，可按需配 GPU** —— 重模型应该跑在 Win 上
- ❌ **我无法直连 Win**（用户硬红线禁 Remote SSH）—— 我推代码，用户拉、跑、报结果
- ⚠️ 模型大小 + 依赖体积 vs 用户安装摩擦（仅 Win 装即可，VPS 可跳过）
- ⚠️ pipeline runtime 速度：hero card 在 hand-start 抓一次/手，1-3s 推理可接受
- ⚠️ §5.2 pytest"全通过"对重模型 inference 路径**无法在 Linux 验证** → 验证标准要拼合"Linux 静态 smoke + Win 用户实测报告"

详见 [[compute-architecture-asymmetry]]。

## 阶段 2：现状分析

### 2.1 当前 codebase 已埋好的"开关"

`recognition/cards.py:32-46`：
```python
def recognize_single(self, image):
    result = self._try_vision(image)   # ← 第 1 优先级
    if result: return result
    result = self._try_heuristic(image)  # ← 第 2 兜底
    ...
```

`recognition/vision.py` 整个文件实现了 **SmolVLM-256M-Instruct** 客户端：
- Apache 2.0 license
- 256M 参数（HuggingFace Hub 上约 500MB 下载）
- 走 `transformers` + `torch`
- CPU 推理可行（用户已有 torch 装好，仅缺 transformers）
- `available` property 自动 fallback：如果 transformers 没装，vision 路径直接返 None，自动走 heuristic

**意味着**：方案 A（启用 SmolVLM）只需 1 行 `pip install transformers` + 首次运行下载模型。零代码改动。

## 阶段 3：方案设计

### 方案 A —— 启用 SmolVLM（recognition/vision.py 已有代码，仅缺 transformers）

**做啥**：
1. `.venv/bin/pip install transformers`
2. 首次运行 SmolVLM 自动从 HF 镜像下载模型到 `.cache/huggingface/`（已在 [[user-machine-topology]] 指向 D 盘项目内）
3. 现有代码自动启用——vision 路径优先，heuristic 兜底

**预期准确率**：80-95%（SmolVLM 是通用 VLM，对"这是什么扑克牌"问题准确率较高；不是专门 poker 但能 zero-shot）

**触发红线**：
- R-1 / R-3 / image-only-compliance：✅ SmolVLM 本地推理 + 截图输入，全合规
- R-8（识别配置一致性）：⚠️ **触发**——新装 transformers 改了模型加载链条；change-log §7 提示用户在 Win 同步装

**优点**：
- ✅ **零代码改动**（已 ready）
- ✅ 准确率大概率从 48% 一跃到 80-95%
- ✅ 同时解决 suit 互窜 + rank OCR 盲点（VLM 看整张图理解）
- ✅ 对亮度变体鲁棒（VLM 训练数据多样）
- ✅ 免费（Apache 2.0，本地推理）

**缺点**：
- ❌ Win 端要装 transformers（额外 50 MB 库）+ 下载 500 MB 模型
- ❌ CPU 推理慢（1-3s/张；hand-start 每手 1 次 = 可接受；但 community 5 张 = 5-15s 可能太慢，留待评估）
- ❌ 首次启动慢（加载模型 5-10s）
- ❌ 增加内存占用（~500 MB RAM）

### 方案 B —— PaddleOCR 替换 EasyOCR

**做啥**：换 OCR 后端到 PaddleOCR（中文 + 数字 OCR 准确率行业领先）

**预期**：rank OCR 提升（7 字体救回），但不解决 suit 互窜问题

**优点**：rank 几乎到 100%
**缺点**：依赖 PaddlePaddle（~1 GB），suit 问题原封不动

### 方案 C —— 自训小 CNN（针对 WePoker UI）

**做啥**：用 31 fixtures + 数据增强（旋转 / 亮度 / 噪声）训一个小 CNN，分类 52 类（13 rank × 4 suit）

**预期**：准确率高（>95%），推理快（<100ms）

**优点**：最快、最准、最专一
**缺点**：要建训练管道（PyTorch 数据增强 + 训练 + 保存）；31 样本偏少；用户每次客户端 UI 变了要重训；DEV 工作量 1-2 天

### 方案 D —— 云 API（Claude / GPT-4V / Gemini Vision）

**做啥**：截图发送到云 API 识别

**触发红线**：**R-3（手牌数据不外传）严重违规** + 可能违反 R-1（视为辅助工具向第三方发送游戏数据）

**结论**：**直接 drop**，不在合规范围内。

### 方案 E —— Template matching（cv2.matchTemplate）

**做啥**：从 fixtures 抽 4 个 suit 的 small pip 模板，runtime 匹配

**预期**：suit 互窜可解；rank 不解

**优点**：零依赖（用现成 cv2）；快（<50ms/张）
**缺点**：模板需手抽 + 对客户端 UI 改动敏感；rank 问题不动

### 方案 F —— Stage 2 继续手搓启发式 + 亮度归一化

**做啥**：HSV V 通道 CLAHE 亮度归一化 + 中央 pip 形状特征（卷积度 / 凸性缺陷）+ template fallback

**预期**：±10-20% 改进，达 60-70%

**优点**：零新依赖；保留对项目栈完全控制
**缺点**：调参时间长（实测 Stage 2 启发式尝试已撞墙）；用户暗态卡天然劣化；7 盲点不解

### 推荐（v2 调整）：**方案 A 启用 SmolVLM，但落地路径改为 Win-only**

**v2 推荐变化**（基于新约束）：
- Linux VPS 不装 transformers / 不下模型（避免 VPS 资源压力）
- Win 端独家装 + 跑 baseline + 报数字
- 代码层 `_try_vision` 在 Linux 自动 fallback 到 None → 走 heuristic（已实现）
- 如果用户后续启用 GPU，inference 时间从 1-3s → < 100ms

**为什么仍然推荐 A**：
1. 零代码改动（已 ready）
2. 预训练 → 不需要训练（无需 Linux dev 训练管道）
3. 一次性跃迁可能性高
4. 可降级（Win 失败时自动 fallback Stage 1 启发式）

### scope 边界（v2 A 启用范围）

- ✅ 做：要求 Win 端用户跑 `pip install transformers`；首次跑下载模型；跑 fixture baseline 看新数字 + 报回；commit + push（仅添加 `requirements.txt` 中 `transformers` 为**可选** dep 注释）
- ✅ 做：在 Linux 写 smoke 测试（确保 `transformers` 缺席时不崩，自动 fallback）
- ❌ 不做：Linux 上装 transformers / 下载模型（避免 VPS 资源）
- ❌ 不做：community 5 张同时识别（暂不评估）
- ❌ 不做：自训 CNN（保留 Stage 3 后备）

### 替代路径：**自训 CNN（Stage 3 后备，需 Win GPU）**

如果 SmolVLM 实测准确率不够（或速度不够），转 Stage 3：
- 我写训练管道（PyTorch DataLoader + 数据增强 + 训练 loop + 保存）
- 用户 Win 端跑训练（可启用 GPU）
- 31 fixture 起步 + 增强 → ~1000 训练样本
- 推理快（<100ms）+ 高准确率（预期 95%+）
- DEV 工作量 1 天（写训练脚本）+ 用户半天（数据准备 + 跑训练）
