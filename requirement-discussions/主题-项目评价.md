## 阶段 1：需求理解与复述

用户要求对 pokemir 项目做中立、客观、允许苛刻的评价。重点不是做下一步实现方案，而是判断这个项目现在到底值不值得继续、强在哪里、短板在哪里、接下来最容易翻车的点是什么。

## 阶段 2：现状分析

### 当前项目状态

- Path A 主线已经从“能识别一点”推进到“真实 Win pipeline 跑、数据写 VPS PG、dashboard 能读并展示”。
- 识别栈有真实 fixture、CNN、摊牌抓帧、OCR fallback，不是停留在 demo 层。
- 数据质量文档里已经承认 P3 fold/check、摊牌物理重复、call/raise 低置信等真问题，说明项目没有把测试通过误当成业务可用。
- 产品形态已经收敛为 Streamlit dashboard + 后续 LLM 解释层，暂时避开 HUD overlay、自动操作等高风险方向。

### 可复用基础设施

- 三机架构：Win 桌面机跑 pipeline，VPS PG 做中转，家用电脑跑 dashboard。
- 契约体系：`contracts/models.sql` / `contracts/api.yaml`。
- 治理体系：REQ / DEV / TEST / DOC 模式、红线、change-log、test-report。
- 数据纠偏入口：`event_corrections` 已独立于 `action_events.raw_data`。

### 主要差距

- 现在还不是“策略助手”，而是“数据采集和质量校准平台”。
- 数据质量仍是最大不确定性；action_events 已有量，但还没完成足够可信的标注闭环。
- dashboard 已能跑，但大部分仍是骨架或 fallback 展示，未形成稳定用户工作流。
- 缺少面向新人的 README，项目入口高度依赖对话记录和 change-log。

## 阶段 3：评价框架

### 评价 A：工程原型质量

项目工程质量高于普通个人原型。它有分层、有测试、有真实数据、有约束、有日志归档，不是“脚本堆起来能跑一次”的玩具。

优点：

- 对不确定性处理比较成熟：fixture、baseline、纠偏表、置信度、raw evidence 都在围绕“别自欺欺人”。
- 架构边界相对清楚：capture / recognition / pipeline / storage / dashboard 分层明确。
- ToS 和隐私边界有红线约束，避免项目朝高风险自动化偏移。

缺点/风险：

- 治理重量明显偏高，很多操作需要写 change-log / report / discussion，个人项目会被流程拖慢。
- 真实业务质量仍未被最终证明；当前最有价值的数据不是“测试通过”，而是长期跑出来的纠偏率和误判分布。
- Windows 真机环境依赖强，复现成本高。

复杂度：中高。

### 评价 B：产品价值

项目方向有真实价值，但价值释放点还没到。现在的核心资产不是 dashboard UI，也不是 LLM，而是持续采集到的、可纠偏的牌局事件数据。

优点：

- 如果数据质量过关，后续 VPIP/PFR/AF、玩家画像、回放复盘、自然语言解释都有清晰承载物。
- dashboard-only 与 pipeline-only 分机运行，让项目可以边采集边开发，工程节奏正确。

缺点/风险：

- 如果 action_events 的基础动作误判率压不下来，后面所有画像都会是“漂亮但不可信”。
- 用户价值链条还长：识别 → 清洗 → 聚合 → 展示 → 决策建议，任何一层弱都会让最终体验像半成品。

复杂度：高。

### 评价 C：长期可维护性

长期可维护性取决于是否能把“治理体系”从负担变成压舱石。现在它能防翻车，但也有把开发变慢的副作用。

优点：

- 关键决策有留痕，跨会话不容易失忆。
- 红线覆盖了 ToS、隐私、schema、ROI、模型路径、全局安装等容易炸的点。

缺点/风险：

- 文档入口不友好：README 缺失，主题文档和 change-log 很多，但新人不知道先看哪 3 个文件。
- 红线已经 10/10 满额，继续加规则会变成“规则替代判断”。
- 有些文件被运行产物污染过，说明 `.gitignore` / 测试副作用还有治理欠账。

复杂度：中。

## 阶段 4：综合结论与推荐

| 维度 | 评价 |
|:---|:---|
| 技术可行性 | 中高；截图 + OCR/CNN + PG + dashboard 路线已经跑通 |
| 产品成熟度 | 中低；还在数据质量与工具化阶段，不是可依赖助手 |
| 工程纪律 | 高；甚至略重 |
| 最大风险 | 数据质量不够可信，导致后续统计和建议失真 |
| 最大资产 | 已经形成真实采集闭环和可纠偏数据流 |
| 下一阶段关键 | 不要急着加 LLM，先把 event baseline / correction loop 做扎实 |

推荐判断：继续做，但必须把“数据质量闭环”放在产品功能之前。这个项目已经过了“能不能跑”的门槛，正在进入“跑出来的数据能不能信”的门槛。后者更难，也更决定项目成败。
