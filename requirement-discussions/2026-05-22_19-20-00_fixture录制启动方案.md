## 阶段 1：需求理解与复述

**用户需求**：从当前 0 张 fixture 到 ≥ 20 张稳定 baseline 的最短可执行路径。

**业务背景**：
- `tests/fixtures/cards/` 目前是空骨架（仅 `.gitkeep` + `_README.md`）+ 1 个 placeholder skip
- 识别精度因此完全没有量化基线，识别相关 bug 修复（`cards.py:109-115` 红黑不分、`_split_card_regions` 边界反转）无法验收
- R-X "识别精度回归保护"红线因此无法激活（README 已埋占位）
- 这是项目当前**唯一阻塞 P0/P1 改进的瓶颈**（评估文档已明确指出）

**核心目标**：让用户在 Win 端用最低操作摩擦把 fixture 数量推到 ≥ 20 张可执行 baseline。

**约束**：
- 真实 poker 客户端截图，不接受合成图（与生产分布脱节）
- R-1 ToS 合规：仅做"截图"，不做键鼠注入 / 进程内存 / 抓包 / 非白名单 Win hook
- R-3 隐私合规：fixture 进 git 仓不含玩家身份信息
- R-9 屏幕捕获合规：限定 ROI 内截图，不调 `capture_raw`

**模糊点**（阶段 5 待澄清）：
- 牌室客户端是哪个（`rois/party_poker.json` 显示 window_title = "Poker Master"，确认是否仍在用？）
- 现有 ROI profile 是否仍准（分辨率 1454×848，客户端版本变了可能失准）
- 录制节奏（一次性 20+ vs 分批攒）

## 阶段 3：方案设计

### 方案 A — 纯手动（Win 截图工具 + 记事本）

**动作**：
1. poker 客户端打开牌桌
2. 每张牌用 Win+Shift+S 框选 + 保存 `<id>.png`
3. 手填 `<id>.json`
4. 拖入 `tests/fixtures/cards/`

**投入**：脚本 0 min；每张约 1-2 min；20 张 ≈ 30-40 min

**风险**：
- 框选区域不一致（每张可能尺寸/位置略不同）→ 与真 ROI 偏差大
- JSON 手填易拼写错（`"h"` vs `"H"`）→ 装载器 assert 失败
- 隐私靠用户自觉（截图可能拍到昵称/聊天）

**红线**：用户自负 R-1/R-3 合规

### 方案 B — 专用录制脚本 `tools/record_card.py`（推荐）

**动作**：
1. 新增 `tools/record_card.py`：
   - 加载 `rois/party_poker.json`
   - 找 poker 客户端窗口
   - 跑循环：用户在客户端摆好牌 → 终端按回车 → 脚本读 ROI（仅 hero_card_1/hero_card_2，**不读 community 因为切分有 bug**）→ 保存 PNG → 提示输入 `rank suit source seq`（或自动 seq）→ 写 JSON
   - 自动按 `<rank><suit>_<source>_<seq>.{png,json}` 命名
   - 单 session 录多张直到用户 Ctrl+C
2. 用户在 Win 端跑：`.\.venv\Scripts\python.exe tools\record_card.py --profile party_poker --source pm`
3. 录到 ≥ 20 张 → `git add tests/fixtures/cards/*.png tests/fixtures/cards/*.json && git commit`

**投入**：脚本 30-60 min（一次性）；每张 ≈ 15-30s；20 张 ≈ 10 min

**风险**：
- ROI profile 失准 → 截到空白或偏离 → 用户肉眼当场可见，可重配 ROI 再录
- 客户端版本/分辨率变了 → 同上
- 用户输入 rank/suit 拼写错 → 脚本输入校验可挡（rank ∈ {2-9,T,J,Q,K,A}，suit ∈ {s,h,d,c}）

**红线合规**：
- R-1 ToS：仅 mss screenshot + ROI 裁切，无键鼠注入/内存抓取/抓包
- R-3 隐私：ROI 限定到卡牌 53×98 / 47×68 像素小区域，物理上不可能拍到昵称/聊天
- R-9 屏幕捕获：调 `capture_roi`，不调 `capture_raw`

### 方案 C — 半自动 recognize-then-confirm

**动作**：B 基础上加一步——保存图后跑 `CardRecognizer.recognize_single` 给出预测 → 终端显示"我猜 Ah，对吗？(y/修正)"

**投入**：脚本多 10-20 min；每张 ≈ 5-15s

**风险**：
- 0 baseline 时识别错率高（要么红黑不分要么 None），用户反而要花精力修正 → 在 fixture ≥ 10 张前价值倒挂
- 用户可能"懒得修"误标 → 污染 fixture

**结论**：等 fixture 攒到 ≥ 30 张、识别 bug 修过后再升 C 才有正回报

### 方案 D — 合成 fixture（用现有 PIL 画卡牌）

**动作**：写脚本随机生成卡牌图像（白底 + 黑/红 rank + suit symbol）→ 直接产出 PNG+JSON

**投入**：脚本 1-2 h；每张 < 1s

**风险**：
- 与真实客户端字体/抗锯齿/光照分布**完全脱节**——fixture 全通生产可能全跪
- 反向证伪 fixture 价值（识别精度数字与真实场景脱节）

**结论**：不推荐作为 baseline 录制路径；可作为补充覆盖（如某 rank-suit 组合真实牌局难凑齐时的 fallback），但**不在本次启动 scope**

## 阶段 4：对比与推荐

| 维度 | A 手动 | **B 脚本** | C 半自动 | D 合成 |
|:---|:---:|:---:|:---:|:---:|
| 一次性脚本投入 | 0 | 30-60 min | 40-80 min | 1-2 h |
| 单张时间 | 1-2 min | 15-30s | 5-15s | < 1s |
| 20 张总耗时 | 30-40 min | **10 min（脚本后）** | 5 min（脚本后） | 1 min（脚本后） |
| 真实性 | ✅ | ✅ | ✅ | ❌ |
| 隐私自动隔离 | ❌ 靠人 | ✅ ROI 限定 | ✅ | N/A |
| 标注错误率 | 高 | 低（输入校验） | 中（0 baseline 信不过预测） | 0 |
| R-1/R-3/R-9 合规可控 | 弱 | 强 | 强 | N/A |

### 推荐：方案 **B**

**理由**：
1. 30-60 min 脚本投入是**一次性**摊销——本次录 20 张 + 未来每次重录/补录都受益
2. ROI 限定从物理上保证 R-3 隐私（不可能拍到昵称）
3. 输入校验天然消除 JSON 标注错误
4. C 在当前 0-baseline 状态价值倒挂；攒到 ≥ 30 张后再做升级
5. 限定**只录 hero 卡**（避开 community 因 `_split_card_regions` 已知 bug 导致的污染），充分单卡 baseline 后再处理 community

### scope 边界（明确不做）

- ❌ 不录 community 卡（等 `_split_card_regions` 修了再说）
- ❌ 不写半自动 recognize-then-confirm（C，留给 ≥ 30 张后）
- ❌ 不写合成 fixture（D，未来 fallback）
- ❌ 不重配 ROI（除非 Q2 确认 profile 失准）
