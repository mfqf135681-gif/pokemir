## 章节 2026-05-28:事件驱动 vs 轮询(三角架构讨论)+ 顺手暴露的 5 个安全盲区

> 状态:**讨论性归档,远期触发**(暂不实施 + 真漏洞另立 #LR13)
> 触发条件:5 秒 polling 让用户感受到痛 / dashboard 同时跑 3 处实例 / 用户分享给 ≥ 2 人需要权限分级

### 1. 用户原始想法

用户提出三角架构假设:**脚本 → DB + dashboard 直推 / dashboard → DB**(并行写),想知道是否合理。用户自评"这个设计问题很大,但说不上来"。

### 2. AI 分析(§9 主动揭盲)

**致命问题(三角双写)**:

1. **双写两份真相** — 同一份数据出现 2 source,后续 audit 灾难(但简单架构里;复杂系统 transactional outbox 可破)
2. **DB = SSOT 原则被打破** — dashboard 持自己副本就脱离 DB 权威(但这是 architectural choice,不是法则)
3. **Streamlit 是无状态 viewer** — dashboard 推数据接哪儿?内存里 5 秒消失;除非加 cache = 第二个非持久 DB

### 3. 用户真问题:**事件驱动 vs 轮询**

AI 识别用户直觉触达的真问题:**事件一发生 dashboard 立刻看到,不等 5 秒 polling**。这跟"动态视觉"(timer 驱动)一脉相承 — 都是事件驱动 vs 轮询的认知。

### 4. 4 个候选方案(归档备查)

| 方案 | 怎么干 | 利 | 弊 |
|:---|:---|:---|:---|
| 🅰️ 缩短 polling 5s → 2s/1s | 简单 | 立即可做 | 浪费 CPU(没新事件也查)|
| 🅱️ PG LISTEN/NOTIFY | 脚本 NOTIFY,dashboard LISTEN | PG 原生,零新组件 | **消息丢失风险**(8KB 上限 / 断线 / PG 重启) — 必须 polling + NOTIFY 互补 |
| 🅲️ Pub/Sub (Redis/RabbitMQ) | broker 中转 | 标准 | 新组件维护 + 新攻击面 |
| 🅳️ WebSocket/SSE | dashboard 长连接 | 真实时 | Streamlit 不天然支持,重写 |

**AI 判断**:🅰️ 简单立即,🅱️ 真路径但有坑。**现在不动**,5 秒 polling 没痛点。

### 5. 顺手暴露的 5 个安全盲区(用户特别关心)

> 用户洞察:"先看是否存在盲区和逻辑漏洞,特别是安全方面" → 这部分比三角设计更值钱。

| # | 盲区 | 严重度 | 处理 |
|:---|:---|:---|:---|
| **A** | **DB 角色权限过宽** — `poker_user` 一个角色 pipeline + dashboard + 工具共用,全表 INSERT/UPDATE/DELETE/TRUNCATE/DROP 权限 | 🔴 R-10 违反 | **真漏洞,立 #LR13**(列入远期,需要时启动) |
| B | dashboard 默认 `127.0.0.1` 只本地;Tailscale 内呢? | 🟠 验证待做 | 加白名单 ACL,需要时启动 |
| C | dashboard 显示 player_name + raw_text 含他人 PII | 🟠 R-3 + ToS 边界 | 加脱敏开关,默认关 |
| D | "脚本 → dashboard 直推"让 Win 端变 listener,攻击面变大 | 🟡 思路问题 | 不实施"直推",从源头避免 |
| E | dashboard INSERT event_corrections 用同一过宽角色 | 🔴 跟 A 是同一漏洞 | #LR13 解决 |

### 6. AI 之前分析的盲区(自我校准)

1. "双写绝对错" — 实际取决于实现,简单架构对,复杂系统有 transactional outbox 等模式
2. "DB = SSOT" — 是选择不是定律
3. "PG LISTEN 是真路径" — oversimplified,实操有坑(详见 §4 🅱️)
4. "实时性 = 价值" — 假设,batch 模式可能也 fine
5. "5 秒 polling 没痛点不动" — 没考虑 dashboard 多实例场景

### 7. 用户决议(2026-05-28)

- **事件驱动 vs 轮询**:列入备选,**触发条件**:5 秒 polling 让用户感受到痛 / dashboard 多实例 / 实时纠偏负担显著
- **DB 角色分离**:**真漏洞** → 列入 #LR13 远期(因为现在自用阶段单角色够,**真要分享时再激活**)
- **PII 脱敏 / dashboard ACL**:同 #LR13,触发条件 = 准备分享 ≥ 2 人
