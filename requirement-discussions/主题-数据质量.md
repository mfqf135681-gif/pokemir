## 章节 2026-05-27:动态视觉折中 + VLM 辅助 ROI(纯思辨)

> 状态:**讨论性归档,远期触发**(不进 P0/P1/P2 落地)
> 触发条件:**WePoker 真改版** OR baseline 校准发现 1Hz 采样不够 OR UI-TARS 类 GUI grounding 模型成熟度有公开 benchmark

### 1. 动态视觉折中方案演进

**起点**:用户提出"不强制把倒计时作主线,改用动态视觉机制 — 哪里动关注哪里 + 注意力分给下一玩家"(类似生物 saccade + attention split)。

**第一轮致命问题**(AI 主动列):
- "动判定" 本身是 OCR/视觉问题,没省到不确定性
- "下一玩家" 判定依赖 state machine,绕不开 timer
- 不是"折中"而是"新方案",复杂度 ≈ timer-burst 完整方案

**第二轮 unlock**(用户提供事实 + AI 自纠):
- **事实**:WePoker 行动玩家有 **亮边 + 倒计时读秒** 两个独立视觉信号
- AI 之前判 "下一玩家循环依赖" 错了 — `_idle_avatar_hash` + `_hamming` 基础设施已在,**头像 hash diff 可替代 timer OCR 作主时序**

### 2. 双信号架构(归档,不实施)

| 信号 | 性质 | 用途 |
|:---|:---|:---|
| **亮边 hash diff** | 静态特征 / 边界事件 | 触发器 — 谁在 act / 结束了 |
| **倒计时读秒** | 动态特征 / 数字滚动 | 心跳 + 时序信号(还剩 X 秒) |

**关键设计:倒计时窗口驱动 burst 节奏**
- T-30s 到 T-5s:低频心跳(1Hz hash check)
- T-5s 到 T-0s:**临界窗口**,切高频 burst(3-5Hz)所有 ROI 全采
- 亮边消失 → 行动完成 → reset
- 倒计时到 -X 秒还没 reset → timebank 触发,继续高频

**附带细化点**:
- 数字位数 transition("10" → "9")时 timer area 像素布局突变 → **免费的"5 秒后必行动"信号,不必 OCR**
- 同时刻只有 1 个 seat 在 act → burst 算力**集中 1 ROI**,不必怕全员高频

### 3. 分层架构 = 抗改版护城河(用户洞察)

用户观点:**这个项目是分层的,UI 改版只影响 ROI + 识别逻辑,不影响后续层**。

```
Layer 1: 捕获 (ROI / capture)        ← UI 改版影响这里
Layer 2: 识别 (OCR / CNN / hash)     ← UI 改版影响这里
Layer 3: 推断 (P3 / Gate / 状态机)   ← 完全无关
Layer 4: 数据落库 + view              ← 完全无关
Layer 5: dashboard / LLM 解释         ← 完全无关
```

**冲击半径估算**:UI 改版后 Layer 1-2 局部修复 ≈ 1-3 天工作量(粗估);Layer 3-5 不动 = **历史数据 / 画像 / 模型权重 carry over**。

**真护城河结论**:**`tools/roi_config.py` 必须保持易用 + 快速重配**(30 min 内完成全 ROI 重框)。这比识别模型多炫更重要。

### 4. VLM 辅助 ROI 重框(=#LR1 vision-assisted ROI 精确落点)

**正确定位**(用户明示):VLM 不进 pipeline 主链路,**仅作 ROI 重框辅助工具** — 改版后用户跑一次 VLM 标注脚本 → VLM 提议 "这里是头像 / 这里是 timer / 这里是 fold 按钮" → 用户人工校正 → 写入 `rois/*.json`。

#### 致命问题(归档时记录)

| # | 问题 | 严重度 |
|:---|:---|:---|
| 1 | VLM grounding(精确 bbox)弱,通用 < 3B 模型对像素坐标能力差 1-2 个数量级 | 🔴 核心瓶颈 |
| 2 | WePoker UI 不在 VLM 训练集,"找 fold 按钮"可能匹配成"任何红色矩形" | 🟠 可补救 |
| 3 | 改版 = 边缘案例 → VLM 在新颖 UI 表现退化,**正在最需要它的时刻最弱** | 🟠 讽刺但真实 |

#### 三个技术路径(归档备查)

| 方案 | 工具 | 强弱 |
|:---|:---|:---|
| 🅰️ | 通用 VLM grounding(Qwen2-VL / GPT-4V) | 强语义,弱坐标 |
| 🅱️ | SAM + CLIP/VLM 语义匹配(切区 + 选语义) | 准 + 慢 + 两阶段 |
| 🅲️ | **UI-TARS / UI-TARS-7B**(ByteDance 2025,**专做 GUI grounding**) | 概念上对症下药,**实测性能未掌握** |

**AI 直觉**:🅲️ 是真路径(GUI-专门预训练),但**没实测**不敢拍胸脯。

### 5. API 方式调用更强模型(GPT-4o / Claude vision / Gemini 等)

**用法定位**:**ROI 重框 PoC 工具 + 一次性配置**(非主链路 / 非长期方案)。

#### 红线评估

| 用法 | R-3(用户自控)| 隐私 / WePoker ToS | 可行性 |
|:---|:---|:---|:---|
| ❌ pipeline 主链路调 API | 严重违反 | — | 不可行 |
| ❌ ROI 重配 + 真实玩家在桌截图 | 违反 | 个人信息流出 | 不可行 |
| 🟡 真实截图 + **本地自动脱敏再上传** | 边缘合规 | — | **鸡生蛋** — 脱敏本身要 ROI 才能定位他人头像 |
| ✅ **空桌 / 测试桌 / 自己单坐桌截图** | 几乎无 PII | 几乎无 | 可行 |
| ✅ **纯 UI 元素截图(不带玩家头像)** | 无 PII | 无 | 可行,但限制使用场景 |

#### 战略评估

- **PoC 价值**:不必本地搭 VLM 环境 → 直接验证 "VLM grounding 这条路可不可行" → 不行立刻放弃,行就考虑本地化迁移
- **理论准确率高 1-2 数量级**(AI 推测,非实测)
- **长期不能依赖 API** — R-3 战略性问题;API 是探路工具,不是长期方案
- **第三方依赖 = 项目稳定性失控**(API 升级 / 价格 / 可用性 / GFW 全在腾讯-OpenAI 手里)
