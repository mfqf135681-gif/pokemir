## 1. 保险 = 什么(WePoker 上下文)

**触发条件**:某玩家 all-in 之后,**对手有 outs 可能赢牌**时,**WePoker 给该玩家(被 all-in 的人)弹保险机会**:
- 玩家可付一笔费用(保险费)→ 若最终 outs 击中 → WePoker 赔付一笔(保险金)
- 玩家也可拒绝 → 正常 all-in 解算

**典型场景**:
- 翻前 / 翻后某人 all-in
- 对手手里有强 outs(如直/同/三条等)
- 系统提示"是否买保险"
- 选 buy → 付保险费,若 outs 击中得保险金
- 选 skip → 正常 all-in

**经济本质**:**WePoker 抽佣的保险产品**,理论上对玩家长期 -EV(WePoker 赚抽佣)。**但短期玩家有买的偏好**(损失厌恶 / 控方差)。

## 2. 当前 pokemir 检测保险的机制

### 2.1 间接推断(已实施,2026-05-25 落地)

代码:`pipeline/orchestrator.py::_infer_insurance`
SQL view:`contracts/views.sql v_player_insurance_stats`

**机制**:hand 结束时,扫描"went-all-in"玩家的 final stack 模式:
- **stack_final = 0** → all-in 输了,**没买保险**
- **stack_final 是非零非整数**(如 137, 245, 1023)→ **买了保险且赢了**(保险金是非整数赔付)
- **stack_final 是整数 round**(如 100, 500, 1000)→ **rebuy 而非保险赔付**

**准确度估计**:中等偏上,但**没有真值校验**。

### 2.2 直接 UI 检测(未实施,远期 #LR8)

WePoker UI 在保险触发时**会显示**:
- 模态弹窗(保险报价 / 买/弃按钮 / 倒计时)
- 保险买后,**赢的那手玩家头上"+xx"前会显示盾牌图标 🛡**
- 可能还有底池区域的"保险费"特殊标识

**阻塞**:
- 自定义俱乐部局**不触发保险**(WePoker 规则)→ 没法主动录样本
- 现场对局观战**完全随机** → 没法定义 ROI
- 没有可控样本 → 无从设计 ROI

→ 列入 [[long-term-roadmap]] #LR8,**等用户能稳定触发保险样本**才启动。

### 2.3 hero 操作面板的"保险"按钮(已排除)

WePoker hero 操作面板右下方有一个**常驻"保险"小圆按钮**(无 all-in 也在)。**用户判定无用**(只是入口,无信息),**ROI 不录**。

## 3. 关联问题汇总(本次 + 历史讨论)

### 3.1 边池布局 vs 单一 pot_size ROI

**问题**:all-in 创建边池(main + side_1[+ side_2])时,布局变化:
- 1 边池:**主+边并排**
- 2 边池:**主+边1 并排,边2 在下方居中**

**影响**:
- pot_size ROI(单矩形)只覆盖 main pot → all-in 手 pot_size_final 失真
- `v_hand_conservation` 守恒律违反多
- 但 **Path B 的 VPIP/PFR/AF 不依赖 pot 总额**,只 BB/100 净胜负依赖

**当前处理**:**接受失真**,Path B 用 `win_amount` 直接读"+xx"算净胜负(2026-05-27 加新 ROI win_amount_area)。

**远期治根**(#LR8 的伴随项):
- 加 `pot_side_1` / `pot_side_2` ROI 字段
- pipeline 求和主+边
- 守恒律对 all-in 局变得精确

### 3.2 保险赔付的盾牌图标

**问题**:玩家买保险后 outs 击中,WePoker 显示"🛡 +xx"(盾牌前缀)在玩家头上方。

**担心**:**win_amount_area** OCR 把盾牌当字符 → 污染 "+xx" 识别。

**解决方案**(2026-05-27 讨论确认):
1. **OCR allowlist 严格模式**:`allowlist="+0123456789"`,盾牌位图自动过滤
2. **双层 fallback**:严格 allowlist 失败 → 宽松 OCR + 正则提取 `\+?\s*(\d+)`
3. **双层防御保险手 OCR 不污染**

**实施**:Path B win_amount 接入 pipeline 时同步做。

### 3.3 自定义俱乐部局不触发保险

**问题**:为了主动录保险 UI 样本,用户曾试自定义俱乐部局触发,**WePoker 配置限制**保险机会不出现。

**结果**:**唯一可获取真实保险 UI 样本的途径 = 真金对局观战**,但**完全随机**(用户无法控制谁 all-in / 是否买保险)。

**当前处理**:接受**保险 UI ROI 没法做**,_infer_insurance 推断兜底。

## 4. 已有数据维度

### 4.1 数据库字段

`hands.raw_data.insurance_inferred`:
```json
{
  "seat_3": {
    "outcome": "bought_and_won",
    "stack_final": 4137,
    "evidence": "non-round stack post all-in"
  }
}
```

### 4.2 SQL view

`v_player_insurance_stats`:
```
columns:
  - player_name
  - all_in_hands            -- 该玩家 all-in 总次数
  - bought_insurance        -- 推断买过保险次数
  - lost_no_ins             -- 没买保险输了次数
  - rebuy                   -- 整数 stack(rebuy 而非保险)
  - insurance_buy_rate_pct  -- 买保险比例
```

### 4.3 当前 stat 价值

可计算:
- "这玩家 all-in 后买保险率 70%" → **极度损失厌恶**,可作画像维度
- "这玩家 all-in 后买保险率 < 10%" → **死硬狂战派**
