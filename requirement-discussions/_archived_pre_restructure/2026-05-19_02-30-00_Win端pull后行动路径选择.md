## 阶段 1：需求理解与复述

**用户需求**：在 Win 测试机上已成功 `git pull` 拉取本会话累积的全部改动（T1+T2+T3+T+1+latest_pot_bb 清零+R-6 ORM 补齐，6 次 commit），询问"下一步该干嘛"。

**业务背景**：
- Win 端在本项目拓扑中扮演**测试机 + 生产 runtime**角色（dev-workflow §4 "Windows 副本只读约定"——Win 不改代码）

**核心目标**：在用户当前时间/精力约束下，选**单一最优行动路径**（不是"全做"），让 Win 端 pull 后的下一步既能拦住潜在回归、又能尽量推进项目目标。

**模糊点**（阶段 5 待澄清）：
- Win 端 `.venv` 是否已装齐当前 `requirements.txt`（本会话未加新依赖，但保险起见需确认）
- Win 端 PG 部署状态（影响 `test_storage` 跑通 vs skip）

- 用户本次精力预算（5-10 min / 30 min / 1-3 h）

## 阶段 3：方案设计

### 方案 A — 最小 sanity 验证（5-10 min）

**动作**：
1. Win 端 `.venv/bin/pip install -r requirements.txt`（保险，验证 deps 同步）
2. `.venv/bin/pytest tests/ -v`
3. 看 Win 端基线（预期 14p/4s/0f 或更多 skip）

**目的**：拦住"Linux 上能跑 Win 上裂"的隐性 bug；零 setup 成本（不动 PG / 客户端 / ROI）

**前置依赖**：L1（venv + deps）

**红线**：无触发；不写代码

### 方案 B — 全链路 pipeline smoke test（20-40 min）

**动作**：
1. 完成方案 A
2. 启动 poker 客户端到牌桌
3. 校验 ROI（`python tools/roi_config.py --verify --name <profile>`）
4. 跑 pipeline 5-10 min（`python main.py` 或等价入口）
5. 验证：mcp__postgres__query / Win PG 客户端 `SELECT * FROM hands` 有数据
6. 观察日志有无 `_process_pot` / `_process_seat_actions` 异常

**目的**：全链路集成验证；揭露 capture → recognize → tracker → DB 任一环节集成 bug

**前置依赖**：L1+L2+L3+L4+L5（全栈）

**红线**：R-9（屏幕捕获）合规需保持；用 `capture_roi` 不用 `capture_raw`

### 方案 C — fixture 录制 baseline（1-3 h）

**动作**：
1. 完成方案 A
2. 启动 poker 客户端到稳定牌桌
3. 用 `tools/roi_config.py` 找出 hero/community 卡牌 ROI
4. 写简单脚本（或手动 mss + cv2 crop）按 ROI 裁出单卡 PNG
5. 标注 `<id>.json`（rank/suit + source + notes）
6. 录 ≥ 20 张覆盖各 rank × suit + 隐私遮挡（dev-workflow §4 例外条款准入 fixtures 双向流转）
7. SCP / git push 回仓
8. Linux dev 上 `bash tools/sync-from-vps.sh`（这次反向）—— 等等，dev-workflow §3 说 Win pull only。**fixture 是例外**（§4 例外条款明示）：Win → push → 仓 → Linux pull
9. Linux 跑 `.venv/bin/pytest tests/test_recognition_fixtures.py -v` 测识别 baseline

**目的**：unlock 后续 `cards.py:109-115` 红黑不分 / `_split_card_regions` 边界反转两个识别 bug 的"可量化修复"路径

**前置依赖**：L1+L2+L4+L5（不需 L3 PG）

**红线**：R-1（牌室 ToS）合规——本地录制 + 隐私遮挡；R-3（手牌数据不外传）合规——fixture 只含卡牌图像，不含玩家身份信息

### 方案 D — 暂时按下 Win 端，转 Linux 做其他独立改动（30 min/任务）

**动作**（任选一条）：
- D.1 R-6 索引同步：把 `contracts/models.sql` 的 3 个 CREATE INDEX 反映到 storage/models.py `__table_args__`
- D.2 latest_pot_bb 跨 hand 防御扫描其它残留状态字段
- D.3 安全收尾——执行 DB 密码轮换（前次会话已写步骤）

**目的**：利用 Win 不在场的时间做 Linux 上能独立完成的事

**前置依赖**：无 Win 端依赖

**红线**：因任务而异（D.1 触发 R-6；D.3 触发 R-2 历史违例收尾）

## 阶段 4：对比与推荐

| 维度 | A 最小 sanity | B 全链路 smoke | C fixture 录制 | D 转 Linux 其他事 |
|:---|:---:|:---:|:---:|:---:|
| 时间投入 | 5-10 min | 20-40 min | 1-3 h | 30 min/任务 |
| 前置依赖深度 | L1 | L1-L5 全栈 | L1+L2+L4+L5 | 无 |
| **验证 pull 没破坏** | ✅ 直接 | ✅ 间接（如全链路过） | ⚠️ 部分（仅识别路径） | ❌ |
| **揭露 OS-specific 回归** | ✅ 单元层 | ✅ 集成层 | ⚠️ 仅识别 | ❌ |
| **推进项目核心目标** | 弱 | 中（验证 pipeline 跑通） | **强**（unlock 识别精度量化） | 弱 |
| **失败定位难度** | 低（变量少） | 高（多层栈） | 中 | 低 |
| **回报/成本比** | **高** | 中 | 中-高 | 低 |

### 推荐：**A → (按用户精力分流) B 或 C**

**理由**：
1. **A 是无条件该做的**——5-10 min 成本，拦住 import / venv / OS-specific 隐性回归；如果 A 暴露问题，B/C 都靠不住（多了 PG/客户端/ROI 几层变量，bug 定位会指数级困难）
2. **A 成功后的分流取决于用户精力**：
   - 当前精力 ≥ 1 h + 有稳定牌桌 → 走 **C**（最高项目目标推进）
   - 当前精力 20-40 min + 有 PG + 牌桌 → 走 **B**（拦截集成 bug）
   - 当前精力 < 20 min → 收工，留 B/C 给下次独立会话
3. **不推荐 D**：Win 已在线，浪费 Win 在场时机做无 Win 依赖的事是机会成本损失

### 兜底（A 失败时）

如果 A 失败：
- 切 DEV 模式排查；不要绕过去做 B/C
- 大概率是 Win 端 venv 缺 dep（`load_dotenv` / `psycopg2-binary` / `easyocr` 等之前加的）
- 或 OS path 差异（虽然代码已用 `pathlib.Path`，但 import 时机的环境变量可能差）
