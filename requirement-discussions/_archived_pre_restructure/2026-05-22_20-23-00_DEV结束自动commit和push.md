## 阶段 1：需求理解与复述

**用户需求**：每次我做完 DEV 任务，**默认我自动 commit + push**；用户 Win 端只需 `git pull`。

**业务背景**：
- 当前流程每次 DEV 结束都让用户做 3 步：1) 看 diff，2) `git add + commit -m`，3) `git push`——用户负担重
- 用户是技术小白，不会主动审 diff，"committing 前看一眼"那一步对他来说价值低
- 用户是单开发者 + 单测试机，没团队代码评审，没 CI，commit/push 不影响他人

**核心目标**：减少 DEV→Win 的循环延迟和操作疲劳。

**模糊点**：
- 我自动 push 出错的代价 vs 现在用户手动的"延迟人审"价值，权衡在哪？
- 哪些情形必须我停手让用户介入？

## 阶段 2：现状分析

### 2.1 现行流程（每次 DEV 任务）

1. 我改文件 + 写 change-log + 跑 pytest
2. 我说"你 commit + push 即可"
3. 用户在 Linux Bash 跑 `git add -A && git commit -m "..." && git push`
4. 用户切到 Win 端 `git pull`

**摩擦点**：
- 用户每次要切到 Linux 终端
- 用户要想 commit message（或抄我给的）
- 用户要等 push 完才能切 Win

### 2.2 自动化的真实风险（不能糊弄过去）

| 风险 | 严重度 | 说明 |
|:---|:---:|:---|
| 误把 `.env` / secret 推上 git history | 🔴 高 | 推上去 = 永久暴露；git filter-repo 清理但 commit hash 全变；R-2 直接踩 |
| 误推未完成的代码 | 🟡 中 | Win pull 后跑出 bug，用户要回头找原因 |
| 误推大文件 / 二进制 | 🟡 中 | git 仓臃肿；GitHub 100MB 单文件硬限会拦但 95MB 能过 |
| 误推 `contracts/` 改动 | 🟡 中 | 数据/API 契约改动按 R-4/R-5 应过 REQ 讨论；自动推绕过这道闸 |
| 误推违反 R-3 的本地数据 | 🟡 中 | 比如真实手牌数据 / DB dump 误进仓 |
| 提交粒度太细 → git log 嘈杂 | 🟢 低 | 美观问题；好的 commit message 能缓解 |

### 2.3 系统提示的约束（外部强约束）

Claude Code 系统提示明确：
> 推送、关闭/评论 PR、发送消息这类**影响共享状态**的动作默认需要用户确认；除非用户在 CLAUDE.md / 持久指令中明确授权，且授权只覆盖明示范围

本讨论 + 落地的 rules-dev 修改正是要建立这种"持久指令"。落地后我的自动 push 行为有了 governance 来源。

### 2.4 用户视角 vs 系统提示约束的妥协空间

- 用户**单人 + 单测试机** → 推送不影响他人
- 但**永久 git history 风险**仍然存在 → 护栏必须设
- 用户不审 diff → 我自己得在 commit 前做 secrets/contracts 检查

→ 结论：可以做，但必须带护栏；不能裸自动。

## 阶段 3：方案设计

### 方案 A — 完全自动（无护栏）

每次 DEV 结束 → `git add <touched files> && git commit -m "<auto-message>" && git push`，无任何前置检查。

- 优点：摩擦最小
- 缺点：所有 2.2 风险全部裸奔
- **不推荐**

### 方案 B — 半自动（带护栏，推荐）

每次 DEV 结束我跑一个"checklist 通过 → 自动 commit + push；任一不通过 → 停手让用户介入"流程。

**护栏 checklist**（任一失败 → 不自动 push）：

1. ✅ 本次 DEV 改的文件清单只含**安全列表**内文件类型/路径：
   - 允许：`tools/*`、`tests/*`、`change-logs/*.md`、`requirement-discussions/*.md`、业务代码（`capture/` `events/` `pipeline/` `recognition/` `storage/`）、`docs/*.md`、`*.gitignore` 改动
   - **禁止自动**（必须用户手动 review）：`.agents/*`（治理规则）、`contracts/*`（数据/API 契约）、`.env*` / 任何 `*.key` / `*.pem` / `*.token`、`docker-compose.yml`、`requirements.txt`
2. ✅ Secrets 扫描：grep 本次 diff 看有没有 `password=` / `token=` / `Bearer ` / 长度 ≥ 32 的 hex 字串（可能是密钥）
3. ✅ 单文件 ≤ 5MB，本次累计变更文件数 ≤ 20 个
4. ✅ pytest 已跑且 0 failed
5. ✅ change-log §1-§9 完整
6. ✅ 当前 git working dir 没有"用户手写的未 commit 改动"（避免误带进我 commit；用 `git status` 检查除我刚改之外的脏文件）

**护栏触发时的 fallback**：
- 我自己跑到 `git add` 那步停下
- 跟用户说"护栏命中 X 项，列出来；要不要你看一眼再决定推不推"
- 用户认可后由用户 push

**commit message 约定**：
- 主标题：本次主要动作（取自 change-log § 1 的"任务概述"首行）
- 后跟 trailer 行：`change-log: <文件名>` + `Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>`

### 方案 C — 保持手动

不动现状。

- 优点：最大用户控制权
- 缺点：摩擦不变

## 阶段 4：对比与推荐

| 维度 | A 全自动 | **B 半自动** | C 手动 |
|:---|:---:|:---:|:---:|
| 用户操作量（每次 DEV）| 0 步 | 0-1 步（仅护栏命中时） | 3 步 |
| Secret 泄漏风险 | 🔴 高 | 🟢 低（扫描挡） | 🟢 低（用户挡） |
| 契约误推风险 | 🔴 高 | 🟢 低（路径黑名单挡） | 🟢 低（用户挡） |
| 提交粒度可控 | 🟡 中 | 🟢 高（每 DEV 一 commit + 清晰 message） | 🟢 高 |
| 系统提示合规 | ❌ 不合规 | ✅ 合规（带护栏 + 明文授权） | ✅ 合规 |
| 实现复杂度 | 简单 | 中等（要写 checklist） | 0 |

### 推荐：**方案 B**

**理由**：
1. 摩擦减到接近 0（绝大多数 DEV 任务护栏不命中，全自动跑完）
2. 风险被护栏全部点名挡掉
3. 用户保留"危险变更必看"的最后一道闸——只在该看时介入
4. 符合系统提示的"durable 授权 + 明示范围"

### 明示授权范围（落地后我会自我遵守）

✅ **授权我自动 commit + push 的范围**：
- 上述方案 B 安全列表内文件
- 通过全部 6 项护栏 checklist
- 在 DEV 模式结束时（不是 REQ / TEST / DOC 模式）

❌ **不在授权范围**（必须仍然问用户）：
- 任何护栏命中的情况
- pytest 失败时
- 用户当前会话明确说"先不要 push" / "本次手动"
- 任何 force-push、amend、reset、revert、删 branch、删 tag 类
- 在 Win 端的任何 git 操作（我不操作 Win 机）
