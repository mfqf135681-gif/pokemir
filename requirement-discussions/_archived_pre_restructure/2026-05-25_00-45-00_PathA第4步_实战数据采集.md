## 阶段 3：方案设计（拆 3 个 sub-REQ + 推荐顺序）

### Sub-REQ 1: Win 端 PG 部署选型

**3 个候选**:

| 方案 | 安装 | 资源 | 维护 | 推荐度 |
|:---|:---|:---|:---|:---:|
| **PostgreSQL 原生 Win installer** | 一个 exe / msi | ~250 MB 磁盘 + 100-200 MB RAM | 简单(服务管理) | ⭐⭐⭐⭐ |
| Docker Desktop + PG container | Docker Desktop(2+ GB)+ docker-compose 1 命令 | Docker 占内存 + PG container | 中等(要装 Docker Desktop) | ⭐⭐ |
| WSL2 + Linux PG | WSL2 + apt install postgresql | WSL2 本身要 8-16 GB 虚拟磁盘 | 复杂(WSL2 + cross-OS) | ⭐ |

**触发红线**:
- R-3:本地 PG,无 R-3 风险
- R-10:Docker Desktop / WSL2 都有"工件出项目目录"问题(更不便管理);**原生 PG 也会写 C:\Program Files\ + C:\ProgramData\**,严格说三者都不完美满足项目内隔离
  - 实操妥协:R-10 主要约束"代码层污染";本地 PG 是系统级 service 装机,与"在 Python 项目内 pipx 全局装"性质不同;接受
- R-2:PG 密码必须 .env,**不能 plaintext 进代码**

**推荐**:**PostgreSQL 原生 Win installer**(选型完整 REQ 落地时再开 sub-REQ)。

### Sub-REQ 2: Seats × 30 ROIs 完整配置

每个 seat 5 个 ROI:
- `action`: 动作文字区域(如 "跟注 100" 显示位置)
- `stack`: 筹码量数字
- `button_indicator`: dealer 按钮位置
- `cards`: 对手底牌区(showdown 时才有内容,大部分情况空)
- `id`: platform user ID 显示区

**6 seats × 5 ROIs = 30 ROIs**。

**触发红线**:R-7 ROI 配置一致性 — 增量配,不改 schema

**推荐**:用 `roi_config.py --field` 一次配一个,或扩展工具支持 `--field seat_N`(批量配单座位 5 个 ROI)。**这个工具增强是 sub-REQ scope 内**。

### Sub-REQ 3: 中文动作 OCR

**两部分**:
1. **模型**:`easyocr.Reader(['en', 'ch_sim'])` — 加中文模型(~50 MB)
2. **代码**:`recognition/actions.py` 加中文 keyword 模式:
   ```
   "跟注" → CALL
   "加注" → RAISE
   "全下"/"All-in" → ALL_IN
   "过牌" → CHECK
   "弃牌" → FOLD
   "下注" → BET
   "下小盲"/"小盲" → POST_SB
   "大盲" → POST_BB
   ```

**触发红线**:R-8 识别模型加载链——加中文模型 → change-log §7 提示 user 模型路径(POKEMIR_EASYOCR_DIR 已重定向 ✓)

**风险**:
- 中文 OCR 性能(~CPU 每次 200-500ms);可能 pipeline tick 变慢
- WePoker 实际动作显示文字可能含数字/英文混排("跟注 100" / "RAISE 200" 等);parser 要兼容

## 阶段 4：对比与推荐

### 推荐路径：**先 Sub-REQ 1 + 3 (一起),后 Sub-REQ 2**

**理由**:

1. **PG + 中文 OCR 是代码 + 装机层,可并行**:
   - 我开 sub-REQ 1 (PG 选型) + sub-REQ 3 (中文 OCR 装机 + actions.py 改) 同步起步
   - 用户:Win 装 PG (我提供指引) + 装 ch_sim 模型
   - 我:改 actions.py 加中文 keyword
   - 完成后**pipeline 可以处理 community / pot / hand 落库**(seat 仍 noop)

2. **Sub-REQ 2 (Seats × 30) 是用户体力活,放最后**:
   - 30 个 ROI 框选是体力,放在 PG + 中文 OCR 都跑通之后做
   - 给用户**有意义的中间产出**(数据已经在落库了),再投入 30 ROI 配置
   - 配完 seats 实战即可拉满

3. **目标分两阶段**:
   - 阶段 A (1+3 完成):pipeline 落库 hands + community + pot;**做不到的:seat actions = 0 行**
   - 阶段 B (+2 完成):seat actions 行有数据;**完整数据采集**

### scope 边界

- ❌ 不包含 Path B(基础统计)— Path B 是 hands 数据上跑 SQL 算 VPIP/PFR;独立讨论
- ❌ 不包含数据可视化 UI— 那是 Path C
