# AGENTS 路由器（通用核 v0.2.1）

> 跨项目通用约束体系的"通用核"层。原则：你看到的每一行 AI 输出，背后都对应一次明确的内部决策落地——**看不见的约束 = 不存在的约束**。

---

## 🚨 最高优先级指令 (Meta-Rule)

每次回复前，必须严格按以下**四步**执行，且**每步结果显性输出**到回复开头。任何静默执行的内部判断视为约束被绕过。

---

### 第一步：意图分类（显性输出 Mode 标签）

每次回复的**第一行**必须按以下两种格式之一输出：

#### A. 初次进入会话 / 用户主动发起新任务

> **[Mode: REQ | DEV | TEST | DOC] | Trigger: User Request** — 一句话归类依据

#### B. 从前一 Mode 切换（Followup）

> **[Mode: Y → X] | Trigger: Followup** — 切换原因（用户语汇 / 上一 task 自然完成）

#### Trigger 定义

| Trigger | 含义 |
|:---|:---|
| `User Request` | 用户主动发起新任务 |
| `Followup` | 上一 task 的延续（如 REQ 完成后用户说"开始实现"）。**仅指当前会话内的延续**——跨会话即使内容相关，仍按 User Request 处理 |

---

#### 决策树（按顺序判断，命中即终止）

1. **用户是否明确要求"讨论方案 / 设计 / 技术选型 / 需求分析"，不要求产出代码？**
   - 关键词：`讨论`、`聊聊`、`设计`、`方案`、`分析一下...怎么做`、`怎么实现比较好`、`技术选型`、`评估`、`建议`、`需求`
   - → **REQ** → 加载 `.agents/rules-req.md`

2. **用户是否明确要求"编写 / 修改 / 删除代码文件"？**
   - 关键词：`实现`、`开发`、`写` + `代码/功能/页面`、`修复` + `改代码`、`新增`、`重构`
   - 且需求本身已足够清晰
   - → **DEV** → 加载 `.agents/rules-dev.md`

3. **用户是否要求"运行 / 测试 / 检查 / 调试 / 复现问题"？**
   - 关键词：`测试`、`跑一遍`、`检查`、`截图`、`抓日志`、`定位`、`这个报错是为什么`、`怎么打不开`
   - → **TEST** → 加载 `.agents/rules-test.md`

4. **用户是否要求"改文档 / 补 doc / 更新 README"？**
   - 关键词：`改文档`、`补 doc`、`更新 README`、`补全 docs/`、`改一下文档`、`文档不对`
   - → **DOC** → 加载 `.agents/rules-doc.md`

5. **以上都不匹配，或同时匹配多个？**
   - 执行下方「复合指令与歧义处理」

---

#### 复合指令与歧义处理

| 场景 | 处理方式 |
|:---|:---|
| **复合指令**（"分析然后实现"、"定位 bug 并修复"、"看完改文档"） | 输出**标准化宣告句**：<br>`本次为复合指令。先执行 [Mode: X] —— 一句目标。完成后等待你确认是否进入 [Mode: Y]。完成标志 = 强制产出物已保存。`<br>识别规则：<br>• 讨论→实现：先 REQ 后 DEV<br>• 测试→修复：先 TEST 后 DEV<br>• 查/看→改文档：先 TEST 后 DOC |
| **分析宾语不明确**（"分析一下 XX 功能"） | • 分析设计/架构 → REQ<br>• 分析运行/错误/性能 → TEST<br>• 无法判断 → **追问**："你希望我分析它的技术实现方案，还是排查它当前的运行问题？" |
| **真正模糊**（"看看这个功能"） | **追问**："你想让我：(A) 讨论方案，(B) 检查当前状态，(C) 直接修代码，(D) 改文档？" |
| **用户回复确认词**（"好的""行""开始吧"） | 查上文：<br>• 上文刚完成 REQ → 切到 **DEV** (Trigger: Followup)<br>• 上文刚完成 TEST → 切到 **DEV** (Trigger: Followup)<br>• 无明确指引 → 追问 |

---

#### Mode 切换纪律

1. **决策树未能命中任一类别时**，默认 = REQ（最安全）
2. AI 不擅自切换 Mode；切换必须用户语汇触发
3. 模式声明优先于内容：用户说"讨论"，即使内容像"开发"，AI 仍保持 REQ
4. REQ Mode 严禁创建代码 / change-log（即使内心觉得"这能直接落地"）
5. **切换必须显式宣告**：从 Mode A 转入 Mode B 时回复第一行必须用 B 格式，并写明切换原因。隐式切换视为违规。

---

### 第二步：动态加载与隔离（显性告知加载文件）

回复的**第二行**必须是：

> 已加载 `.agents/rules-X.md` 作为唯一行为准则（X = req | dev | test | doc）

#### 隔离规则

- 🔒 **行为规则文件互斥**：`rules-req.md` / `rules-dev.md` / `rules-test.md` / `rules-doc.md` —— 四者**绝不混读**
- 🔓 **参考数据共享**：`contracts/` / `docs/` / 项目代码 / `requirement-discussions/` / `change-logs/` / `test-reports/` —— 任何 Mode 均可按需读取
- 🔓 **通俗化约束跨模式生效**：`.agents/communication.md` —— **所有 Mode 强制遵守**，不计入互斥范围

---

### 第三步：项目红线显性核验（基于 project-constraints.md）

读取 `.agents/project-constraints.md`：

#### 缺失处理（两级警告策略）

- **首次发现缺失**（项目刚挂载）：回复中显式提醒一次 `⚠️ 项目未配置 project-constraints.md，建议在 REQ 模式下创建红线清单`，并在 Router 第四步自检中标 `□ 已建议用户创建 project-constraints.md`
- **后续回复中**：标注降级为一行简注 `[Layer 1 未配置]`

#### 占位符识别

红线条目含 `__PLACEHOLDER__` 字符串 → 视为未实际定义，**整条跳过核验**（不输出"R-X (✓ 未触发：__PLACEHOLDER__)" 这种噪音）。

#### 已配置红线的核验

回复的**第三行起**输出红线核验结果，格式：

> **红线核验**：
> - **触发**：
>   - R-N（✗ 触发：[一句话事实，引用具体改动路径/关键字]）
>   - R-M（✗ 触发：...）
> - **已逐条核验未触发**：R-X, R-Y, R-Z, ...

#### 强制约束

- ⚠️ **触发项必须独立列出 + 附事实理由**（"改了 X 文件" / "调用了 Y 函数"），不允许只写"触发"
- ⚠️ **未触发项必须以 ID 列表形式合并显示**——禁止省略 ID 写"其他未触发"或"无关红线"
- ⚠️ **本任务无任何触发** → 仅写一行 `已逐条核验未触发：R-1, R-2, ...`，不写"无"
- ⚠️ **红线触发条件中任何一项命中即整条红线触发**，不论命中范围大小。合规动作应覆盖全部涵盖范围（不只命中部分）
- ⚠️ **红线总数上限 10 条**：超过即提示用户切到 REQ 模式讨论收敛红线
- ⚠️ **后续输出必须用 R-X 引用**：change-log / test-report / req-discussion 中提及红线必写 ID

---

### 第四步：任务完成自检清单（显性打印）

任务结束前必须**显性打印**以下 checklist，每条标注 ✅ / ❌ / N/A。**禁止静默勾选**。

| Mode | 强制产出物 |
|:---|:---|
| **REQ**（完整讨论） | □ `requirement-discussions/YYYY-MM-DD_HH-mm-ss_<主题>.md` 已保存？<br>□ 已遵守通俗化三件套（❓问什么 / 🎯选项+预测 / 💡推荐建议）？ |
| **DEV** | □ `change-logs/YYYY-MM-DD_HH-mm-ss_<任务简述>.md` 已保存？<br>□ 测试套件已运行且**全部通过**？<br>□ 触发的红线 ID 已在 change-log 显式引用？<br>□ 关联 requirement-discussion 已 cross-link（功能/契约改动场景）？ |
| **TEST**（完整诊断） | □ `test-reports/YYYY-MM-DD_HH-mm-ss_[状态]_<简述>.md` 已保存？<br>□ `test-reports/INDEX.md` 已同步？<br>□ 截图/日志证据已归档？ |
| **DOC** | □ 仅修改了 `docs/` 或 `README.md` / `CHANGELOG.md`？<br>□ 已输出 diff 摘要（每文件一行：路径 + 一句变更说明）？ |

#### 违规处置

任一项 ❌ 且任务**实际推进了对应产出** → **不得回复"任务完成"**，必须执行以下三步：

1. **先补齐缺失项**——回到对应规则文件重新执行
2. **在归档对象顶部显式标注违规**——格式：
   ```
   ⚠️ 本次违规：[第 N 步未按规则执行] —— [原因] —— [补救动作]
   ```
   写入 `change-log` / `test-report` / `requirement-discussion` 的元信息区
3. **下次任务开始时主动复盘**——在意图分类标签后追加 `(上次违规已修正)`

> 不靠 AI 自觉，靠归档对象留痕——用户可以通过翻阅 `change-logs/` 等目录直接审计违规历史。

#### 快速路径例外

如本次为各 rules-X.md 中定义的"**快速**"分支（快速验证 / 快速诊断 / 快速讨论 / 快速改文档），可跳过对应产出物文件，但：
- **自检清单仍需显性打印**
- 跳过的项**必须连带写**：`N/A — 触发 [Mode] 快速路径，依据：[四条件中具体哪一条]`

## 通俗化输出（贯穿所有 Mode）

详见 `.agents/communication.md`。**所有 Mode 强制生效**，包括但不限于：

- **首句一句话讲人话**：每个分析点、方案、待确认项的**第一句**不出现项目术语
- **术语首次出现必释**：紧跟 6-15 字口语化解释（可查 `.agents/project-glossary.md`，如有）
- **影响翻译为用户可感知现象**：不说"违反引用完整性"，要说"页面会出现一个连不上数据的孤立条目"
- **待确认项必含三件套**：❓ 问什么 / 🎯 选项 + 预测 / 💡 推荐建议
- **数字必须有上下文**：不说"延迟 30%"，要说"列表打开多等 200ms"

**自检**：写完后通读一遍——**完全不懂代码的产品经理读完仍不知道选 A 还是 B？必须重写。**

## ⚠️ 重要提醒

- **本路由器文件 (`AGENTS.md`) 仅用于决策。** 不要直接应用本文件中的任何开发、测试、需求讨论、文档规则。
- 所有具体约束必须从对应的 `.agents/rules-X.md` 中读取。
- **遇到无法归类的输入，宁可追问，不可猜测。**
- **四步流程不可压缩**：意图分类 → 加载隔离 → 红线核验 → 自检清单。任一步骤跳过即构成违规。
- **可见性是核心审计标准**：任何静默执行的内部判断都视为约束被绕过。
- **本协议属于 Layer 0 通用核**：跨项目同形。项目特化内容只能写入 Layer 1 文件（`project-constraints.md` / `project-glossary.md`），**严禁污染 Layer 0 文件**。
