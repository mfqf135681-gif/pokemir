## 3. 关键技术决策记录

### 3.1 数据获取路径选择
- 走过的弯路：① 真实下注桌 hero 卡（用户分散精力）② 观战 showdown 看别人底牌（免费次数限制）
- 最终方案：**观战 + 录公共牌 flop 3 张**（每手稳定 3 张,免费,无 ToS 风险）

### 3.2 模型选型路径
- 评估过：SmolVLM (256M VLM)、PaddleOCR、模板匹配、自训 CNN
- 决议：**Stage C 自训 CNN**（用户 5070 Ti 算力 + 准确率 + 延迟综合最优）
- 拒绝理由：
  - SmolVLM/Qwen2-VL: 推理 200-500ms 太慢
  - PaddleOCR: 解决 rank 但不解决 suit
  - 模板匹配（DeepSeek 建议）: WePoker 暗态卡 + 窗口变化使其不可靠
  - H100 等更强 GPU: 数据/模型容量瓶颈,算力远远过剩,投资 0 ROI

### 3.3 算法关键参数调优历程
- rare-weight = 5.0 → 暴露"中间地带忽视"副作用（Ts/Th 5 张样本反不如 2 张时学得好）
- 用户精准诊断"问题不在样本量"
- 调到 **rare-weight = 2.5** → T 类回归 + rare 类仍有适度加成 → 100% val

### 3.4 标签错误教训
共发现 3 处人工标签错（5h_pm_002, Jh_pm_003, 4s_pm_005）。规律：
- 用户在长时间录制 / 复检时易输错 rank 或 suit
- record_card.py 完全相信用户输入,无二次校验
- 部分 commit 流程中 git mv + edit 容易漏 stage 后续 edit（commit 0c217bc 教训）

## 4. 最终架构

```
recognition/cards.py CardRecognizer.recognize_single():
    1. _try_cnn()         ← 主路径：自训 CNN (models/card_cnn.pth)
    2. _try_vision()      ← 后备：SmolVLM (transformers 装则启用,否则跳过)
    3. _try_heuristic()   ← 兜底：OCR allowlist + 多 crop + 颜色判定
    4. None               ← 全失败时返回 (不阻塞 pipeline)
```

CNN 模型不在 git（`models/` gitignored）, 用户 Win 端跑 `tools/train_card_cnn.py` 产出。
