## 1. 任务概述

- **用户原始需求**：Win pull 等待期间的 Linux 小修补 2/2——补齐 ORM
- **drift 描述**：`contracts/models.sql` 定义 5 张表，`storage/models.py` 只定义 2 个 ORM class——`player_stats_cache` / `player_situational_stats` / `replay_corrections` 缺失。后果：
  - `init_db()` 调 `Base.metadata.create_all()` 只能创建 2 张表 → PG 实际只有 2 张表（T0 smoke test 已确认）
  - 未来任何想往这 3 张表读写的代码会因为缺 ORM class 而无法用 SQLAlchemy 操作
- **涉及功能模块**：`storage/models.py`（ORM 定义）；间接：`init_db()` 行为
- **R-6 方向**：本任务**反向**——契约 SQL 不变，让 ORM 追上契约（标准 R-6 触发是契约改→ORM 跟改；本次是修历史 drift）

## 2. 假设清单

| # | 假设 | 出处 |
|---|---|---|
| 1 | 不引入 Alembic migration（仅用 `create_all`） | 项目当前 storage/database.py 用 `Base.metadata.create_all()` 模式；3 张表尚无数据；引入 Alembic 是独立基建任务 |
| 2 | 不加 Repository class 与单元测试 | rules-dev §3 最小实现——尚无消费方代码；提前加 Repository 会成死代码 |
| 3 | NOT NULL + DEFAULT 列用 `server_default` 而非 Python `default` | 契约规定 PG 级 DEFAULT；Python-side default 仅在 SQLAlchemy ORM 路径生效，外部客户端（psql、未来微服务）INSERT 不带该列会触发 NOT NULL violation |
| 4 | 索引（`idx_sit_stats_player` / `idx_sit_stats_type` / `idx_corrections_hand`）暂不在 ORM 声明 | R-6 强约束是表/列结构同步；索引是性能层，可后续单独同步；当前 3 张表无消费方，性能 N/A |
| 5 | 物理表需 drop + 重建：上次 create_all 已建了 3 张表（Python-side default，无 PG DEFAULT）→ 需 drop 后让 init_db 用新 server_default 重建 | mcp__postgres__query 验证 3 张表均空（0 行）；drop 无数据损失 |
