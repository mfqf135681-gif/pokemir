# AGENTS 路由器 (Router)

## 最高优先级指令 (Meta-Rule)

在回答用户任何请求之前，你必须严格按以下流程执行：

### 第一步：意图分类 (Intent Classification)

分析用户当前输入，将其归类为以下三者之一。

#### 分类依据（按以下决策树顺序判断）

1. **用户是否明确要求"讨论方案 / 设计 / 技术选型 / 需求分析"，而不要求产出代码？**
   - 关键词：`讨论`、`设计`、`方案`、`分析一下...怎么做`、`怎么实现比较好`、`技术选型`、`评估`、`建议`、`需求`
   - → **REQUIREMENT_DISCUSSION** → 读取 `.agents/rules-req.md`

2. **用户是否明确要求"编写 / 修改 / 删除代码文件"？**
   - 关键词：`实现`、`开发`、`写` + `代码/功能/页面`、`修复` + `改代码`、`新增`、`重构`
   - **且**需求本身已足够清晰（不需要先讨论方案）
   - → **DEVELOPMENT** → 读取 `.agents/rules-dev.md`

3. **用户是否要求"运行 / 测试 / 检查 / 调试 / 复现问题"？**
   - 关键词：`测试`、`跑一遍`、`检查`、`截图`、`抓日志`、`定位`、`这个报错是为什么`、`怎么打不开`
   - → **TESTING** → 读取 `.agents/rules-test.md`

4. **以上都不匹配，或同时匹配多个类别？**
   - 执行下方的「复合指令与歧义处理」

#### 复合指令与歧义处理

| 场景 | 处理方式 |
|:---|:---|
| **复合指令**（如"分析然后实现"、"定位 bug 并修复"） | 识别指令中的**第一步**是什么：<br>• "分析/讨论 → 实现" → 先走 **REQUIREMENT_DISCUSSION**<br>• "定位/测试 → 修复" → 先走 **TESTING**<br>• 告知用户："我先进行 [第一步]，完成后你再确认是否进入下一步。" |
| **"分析"宾语不明确**（如"分析一下 [某功能]"） | • 分析**设计方案/架构** → REQ<br>• 分析**运行行为/错误/性能** → TEST<br>• 无法判断 → **追问用户**："你希望我分析它的技术实现方案，还是排查它当前的运行问题？" |
| **真正模糊**（如"看看这个功能"、"这个功能"） | **不静默归类。追问用户**："你想让我：(A) 讨论这个功能的设计方案，(B) 检查它当前的运行状态，还是 (C) 直接修改代码？" |
| **用户回复确认词**（如"好的"、"行"、"可以"、"就这样"、"开始吧"） | 查看对话上文：<br>• 若上文刚完成 REQ 讨论 → 视为切换到 **DEVELOPMENT**<br>• 若上文刚完成 TEST 诊断 → 视为切换到 **DEVELOPMENT**<br>• 若无上文 → 追问用户意图 |

### 第二步：动态加载与隔离 (Dynamic Loading & Isolation)

1. **一旦完成分类，立即读取对应的规则文件。**
2. **将读取到的内容作为你接下来的唯一行为准则。**
3. **隔离范围说明：**
   - 🔒 **行为规则文件互斥**：`.agents/rules-req.md`、`.agents/rules-dev.md`、`.agents/rules-test.md` —— 三者绝不混读。
   - 🔓 **参考数据文件共享**：`contracts/`、`docs/`、项目代码、`requirement-discussions/`、`change-logs/`、`test-reports/` —— 任何模式下均可按需读取，不受隔离限制。

---

### ⚠️ 重要提醒

- **本路由器文件 (`AGENTS.md`) 仅用于决策。**
- **不要直接应用本文件中的任何开发、测试或需求讨论规则。**
- 所有具体约束必须从对应的 `.agents/` 子文件中读取。
- **遇到无法归类的输入，宁可追问，不可猜测。**
