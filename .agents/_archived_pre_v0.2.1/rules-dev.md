# 开发代理执行规则

## 1. 适用范围

本规则集适用于开发阶段的代码编写与修改任务。

权威来源（只读——禁止修改）：
- `docs/` — 系统文档
- `contracts/api.yaml` — API 契约
- `contracts/models.sql` — 数据模型契约
- `tests/` — 测试套件
- `requirement-discussions/` — 需求讨论记录（`confirmed` 状态为权威依据）

> 注：本项目当前无独立的前端 UI 设计规范文件；HUD 层进入 Phase 4-5 后再建立。

---

## 2. 实现前规则

编写任何代码之前：
- **查阅需求讨论记录。** 在 `requirement-discussions/` 目录下搜索与本次任务相关的讨论记录。
  - 若存在 `confirmed` 状态的记录 → **必须**将其作为实现依据，在「假设清单」中引用具体文件路径。
  - 若存在相关但状态为 `pending` 的记录 → 提醒用户先完成需求确认再动手。
  - 若无相关记录 → 继续执行下方规则，但留意是否存在需要先讨论的方案盲区。
- **列出所有前提假设。** 每条假设必须标注出处，无法确认来源的标记为 `UNKNOWN`。
- **不确定即标记 UNKNOWN。** 禁止猜测字段名、响应体结构或业务逻辑。
- **遇歧义即停止。** 若需求存在多种合理解读，立即暂停并请求澄清——禁止默认选择某种解读。
  - 若歧义涉及**整体方案方向**而非具体实现细节，建议用户先进入 **REQUIREMENT_DISCUSSION** 模式（`.agents/rules-req.md`）讨论清楚后再回来实现。
- **禁止隐式假设。** 每个决策必须可追溯至契约、需求讨论记录、文档或用户的明确指令。若四者均无依据，停止。
- **需求不清晰 → 停止。** 输出不清晰之处、所需信息及应向谁确认。
  - 若不清晰之处属于"功能怎么做"级别的方向性问题，主动建议：
    > "这个问题涉及方案选型，建议先进入 **REQUIREMENT_DISCUSSION** 模式讨论。你可以说'讨论一下 [需求] 的设计方案'来触发。"

---

## 3. 实现规则

编写代码时：
- **只实现当前任务。** 禁止添加未被要求的功能、工具函数、抽象层或配置。
- **禁止投机性扩展。** 禁止"顺手"重构、过早泛化、"预留"的可选参数。
- **最小实现优先。** 选择满足需求的最小改动。
- **遵守契约。**
  - 所有请求/响应体结构必须与 `contracts/api.yaml` 一致。
  - 所有数据写入必须与 `contracts/models.sql` 一致。
  - 禁止发明契约中未定义的字段、集合或响应属性。
  - 禁止向客户端返回契约中未记录的字段。
- **保持现有模式。** 与周围模块的代码风格、命名习惯和结构保持一致。
- **遵循设计规范。** 创建或修改前端 UI 时，若设计规范文件存在则遵循其规范；当前项目无前端规范文件，跳过本条。
- **响应格式。** 遵循项目既定的响应格式约定（如统一包装器或错误处理模式），新增接口与现有接口保持一致。

## 4. 修改规则

修改已有代码时：
- **精确范围。** 只触碰当前任务直接涉及的文件和代码行。
- **禁止无关重构。** 禁止对任务边界之外的代码进行重命名、重组织或"优化"。
- **禁止外观修饰。** 未发生功能变更的文件中，禁止调整空白、改写注释、重排 import。
- **保留历史代码。** 除非用户明确要求，否则禁止删除已有代码——即使看起来未被使用。
- **只清理自己产生的冗余。** 仅可移除当前任务中引入的死代码。
- **若修改会影响以下内容，立即停止并报告冲突：**
  - `contracts/api.yaml` 中定义的 API 签名或响应体结构
  - `contracts/models.sql` 中定义的数据模型
  - 被其他模块消费的公共接口
  报告冲突后等待指令。
  - **例外**：若 `requirement-discussions/` 中的 `confirmed` 记录已明确确认了上述变更，按记录执行，并在变更日志中引用该记录。
- **共享模块同步。** 本项目目前无被多模块复制的共享代码，此条自动忽略。

## 5. 验证规则

每次实现完成后：

### 5.1 判定：完整验证 vs 快速验证

| 条件 | 走哪条路径 |
|:---|:---|
| 满足以下**全部**条件：<br>① 改动仅涉及**单个文件**<br>② 改动行数 **≤ 20 行**<br>③ 不涉及**接口签名变更**、**数据库 schema 变更**、**公共模块变更**<br>④ 改动类型为**纯 UI 调整**（文字、样式）或**单行逻辑修正** | **快速验证**：手动确认改动生效即可。在变更日志中标注"轻量改动，手动验证通过"。 |
| 以上任一条件不满足 | **完整验证**：执行下方完整流程。 |

### 5.2 完整验证流程

- **开始前定义可验证目标**（例如："页面渲染正常"、"API 返回正确响应"、"测试 X 通过"）。
- **执行测试套件：**
  ```
  pytest tests/ -v
  ```
  > 部分测试需要外部依赖：`test_storage.py` 需要本地 PostgreSQL（按 `config.py` 默认 DSN）；`test_capture.py` 需要可用屏幕/窗口。无相应依赖时这些用例会 skip，不算失败。
- **所有测试必须通过。** 若任何测试失败：
  1. 诊断失败原因。
  2. 修复实现代码（而非修改测试）。
  3. 反复执行，直至全部通过。
- **例外——经批准的契约变更。** 若测试失败的唯一原因是在同一任务中经用户明确批准修改了契约，或在 `requirement-discussions/` 的 `confirmed` 记录中已确认契约修订，允许更新测试以匹配新契约。须在任务输出中记录变更了哪条测试及原因。
- **禁止在测试未全通过时结束任务。** 测试套件标红是阻塞性错误，不可视为已知问题搁置。

## 6. 契约强制约束

硬性约束——违反任何一条均为阻塞性错误：

| 规则 | 详细说明 |
|------|----------|
| 禁止未声明的 API 调用 | 每个对外接口调用必须在 `contracts/api.yaml` 中有对应定义。 |
| 禁止未声明的字段 | 请求体和响应体只能包含契约 schema 中定义的属性。 |
| 禁止未声明的数据存储 | 数据库读写只能指向 `contracts/models.sql` 中定义的结构。 |
| 禁止临时数据结构 | 禁止创建与契约 schema 矛盾或超出其定义的内联对象结构。 |
| 模式变更须上报 | 若任务确实需要新字段或新端点，停止并请求契约修订——禁止自行添补。例外：若 `requirement-discussions/` 中 `confirmed` 记录已确认修订，按记录执行。 |

## 7. 异常处理

出现问题时：

| 场景 | 要求的操作 |
|------|-----------|
| 对需求存在不确定 | **停止。** 输出不确定之处。若属于方案方向性不确定，建议进入 REQUIREMENT_DISCUSSION 模式。 |
| 契约未覆盖当前场景 | **停止。** 报告缺口，禁止自行发明数据结构。建议先进入 REQUIREMENT_DISCUSSION 讨论方案。 |
| 修改后测试失败 | **修复。** 反复迭代直至测试全部通过。 |
| 文档与契约冲突 | **停止。** 报告差异。契约具有最高权威性。 |
| 实现会破坏其他模块 | **停止。** 报告影响范围。 |
| 在规则约束下无法完成任务 | **停止。** 说明阻塞原因，禁止放宽约束。 |

绝不为推进进度而绕过约束。暂停的任务优于损坏的系统。

---

## 8. 允许与禁止事项（速查）

| ✅ 允许 | ❌ 禁止 |
|:---|:---|
| 修改本次任务直接涉及的文件 | 修改任务边界外的代码 |
| 创建新文件（被任务要求时） | 删除未被要求删除的已有代码 |
| 读取所有项目文件、契约、文档、讨论记录 | 修改 `contracts/` 下任何文件而无 confirmed 讨论记录授权 |
| 轻量改动走快速验证路径 | 在测试未全通过时结束任务 |
| 执行 REQ 讨论中确认的契约修订 | 自行发明契约中未定义的数据结构 |
| 在变更日志中引用讨论记录 | 猜测字段名、响应体或业务逻辑 |

## 9. 手动操作提醒

部分变更需要开发者在代码库之外执行手动步骤。完成任务后，检查是否涉及以下情形，若涉及则**输出清晰的 ⚠️ 手动操作 区块**，逐项列出所需步骤：

| 触发条件 | 所需手动操作 |
|----------|-------------|
| 代码中引用了新的数据表 | 在 PostgreSQL 中执行对应的 `CREATE TABLE` 或 Alembic migration |
| 修改了 `contracts/models.sql` | 同步更新 `storage/models.py` ORM 定义，必要时生成迁移脚本 |
| 修改了 `requirements.txt` | 提示用户执行 `uv pip install -r requirements.txt` |
| 新增了环境变量 | 同步更新 `.env.example` 并提示用户更新本地 `.env` |
| 修改了 ROI 配置结构 | 提示用户重新运行 `python tools/roi_config.py --name <profile>` |
| 修改了识别模型 / Vision client | 提示用户确认 `POKEMIR_MODEL_DIR` 或 `HF_ENDPOINT` 配置一致 |

格式：
```
⚠️ 手动操作：

[具体步骤]
[具体步骤]
```

若无需手动步骤，不输出此区块。

## 10. 修改日志

每次完成代码修改任务后，必须生成一份结构化的修改日志并保存到 `change-logs/` 目录。

### 10.1 日志存储位置

所有修改日志必须保存到：
```
change-logs/
```

若目录不存在，请先创建目录。

### 10.2 文件命名规范

```
YYYY-MM-DD_HH-mm-ss_<任务简述>.md
```

示例：`2026-04-22_16-20-33_修复登录token签发.md`

### 10.3 日志内容结构

日志文件必须包含以下章节，使用 Markdown 格式：

#### 1. 任务概述
- 用户原始需求摘要
- 涉及的功能模块
- **关联需求讨论记录**：`requirement-discussions/xxx.md`（如无则写"无"）

#### 2. 假设清单
列出本次实现所依赖的所有前提假设及其出处（来自 `2. 实现前规则` 的要求）。

#### 3. 文件变更清单

| 文件路径 | 变更类型 | 变更说明 |
|----------|----------|----------|
| `path/to/file.py` | 修改 | 修复了... |
| `path/to/newfile.py` | 新增 | 增加了... |

#### 4. 契约一致性检查
- 是否涉及 `contracts/api.yaml` 或 `contracts/models.sql` 变更（是/否）
- 如是，说明变更内容及批准情况（用户直接批准 / REQ 讨论记录确认 / 其他）

#### 5. 测试结果
- 验证路径：完整验证 / 快速验证
- 执行的测试套件命令或手动验证步骤
- 测试通过情况（全部通过 / 部分失败及原因）

#### 6. 手动操作提醒
若任务触发了第 9 节中的任何手动操作条件，在此逐项列出。

#### 7. 潜在影响范围
- 简要评估本次修改可能影响的其他模块或功能

### 10.4 保存动作

在完成第 9 节的手动操作检查后，必须：
1. 将以上内容汇总为一份 Markdown 文件。
2. 将日志写入 `change-logs/` 目录。
3. 向用户确认：
```
✅ 修改日志已保存至：change-logs/YYYY-MM-DD_HH-mm-ss_<任务简述>.md
```
