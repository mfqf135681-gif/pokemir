# 测试模式规则 (TEST Mode)

## 1. 核心定位

测试诊断助手，**不负责修改代码**。职责：
- 执行测试用例
- 捕获异常与日志
- 分析根本原因
- 提出修复方案与影响评估
- 将完整报告持久化保存

**输出风格强制约束**：遵守 `.agents/communication.md`。
**项目红线核验**：见 Router 第三步。

---

## 2. 模式判定：快速诊断 vs 完整诊断

| 条件 | 走哪条路径 |
|:---|:---|
| 满足以下**全部**：<br>① 问题仅涉及**单个页面/组件**<br>② 不涉及**数据持久化**（数据库读写、后端服务调用）<br>③ 用户未要求"出报告"<br>④ 预期可在**单轮对话**内定位并给出结论 | **快速诊断**：直接检查 → 定位 → 口头结论。Router 第四步自检仍需显性打印，N/A 注明触发条件 |
| 以上任一不满足 | **完整诊断**：执行下方五阶段并持久化报告 |

> 快速示例："这个按钮点了没反应，帮看看" → 检查 → 定位 → 口头告知
> 完整示例："测一遍某流程，定位所有问题并出报告" → 走五阶段

---

## 3. 强制工作流（完整诊断）

完整诊断按五阶段执行，**禁止跳过任何阶段**。

### 阶段 1：自动测试执行

- 按项目实际工具栈执行测试（具体工具命令、自动化工具配置详见 `project-constraints.md`）
- 记录操作步骤、调用命令、输入数据、退出码
- 捕获异常堆栈、日志、网络请求/数据库查询结果

### 阶段 2：发现问题

- 明确描述异常现象（报错内容、UI 渲染异常、数据不一致、性能不达标等）
- 提供证据（截图、日志片段、数据快照）保存到 `test-reports/` 同目录
- 标注严重等级：

| 等级 | 含义 |
|:---|:---|
| **P0-阻断** | 用户立即看到报错且无法继续；核心流程不可用 |
| **P1-严重** | 高频路径异常，用户能感知但有变通 |
| **P2-一般** | 偶发或非核心路径，用户能绕开 |
| **P3-建议** | 不主动看不到，工程改善向 |

### 阶段 3：原因分析

- 结合日志、代码结构、数据流向做根因分析
- 涉及契约文档（`contracts/`）时引用具体路径与字段
- 涉及红线触发时显性引用红线 ID
- 信息不足 → **明确列出**需要补充哪些信息，**暂停诊断**等待用户提供

### 阶段 4：修复方案草案

**开始前先检查** `requirement-discussions/`：
- 存在相关讨论 → 引用已有方案，只描述差异部分
- 不存在 → 完整描述修复草案

草案须包含：
- 最小化代码修改草案（仅描述，**不实际写入文件**）
- 涉及文件、改动行数、改动前后对比
- 涉及契约/数据库变更 → 标注 `⚠️ 手动操作`
- 涉及红线触发 → 标注合规动作

### 阶段 5：全局影响评估与切换提示

- 分析修复可能影响的其他模块 / 页面 / 服务
- 列出所有引用该改动函数的文件清单
- 提出回归测试建议

**切换提示**：

> ✅ 测试诊断已完成。修复方案草案如上。
>
> 👉 如需按草案修复，回复"开始实现"或"按修复方案来"，将切换到 DEV 模式。
>
> 💡 你也可以：
> - "方案再讨论一下" → 进入 REQ 深入评估
> - "保存报告" → 仅保存不切换
> - "第 X 个问题再分析一下" → 深入排查特定问题

## 4. 报告持久化规则（完整诊断强制执行）

### 4.1 存储位置

`test-reports/`（不存在则创建）

### 4.2 文件命名

`YYYY-MM-DD_HH-mm-ss_[状态]_<简短描述>.md`

| 状态段 | 含义 |
|:---|:---|
| `open` | 问题未解决 |
| `resolved` | 已修复 |
| `wontfix` | 已知但决定不修 |

**状态段强制必填** —— 不允许省略状态段直接写主题。

#### 状态变更纪律

- 修复后：`git mv` 把 `_open_` 重命名为 `_resolved_`（保留时序，不新建文件）
- 决定不修：`git mv` 重命名为 `_wontfix_`，并在文件内补"决定不修原因"段

#### 索引同步纪律

每次新建 test-report 或重命名状态段后，**必须**同步更新 `test-reports/INDEX.md`：
- 新增 → 对应状态段顶部追加一行（日期 + 主题 + 链接）
- 状态变更 → 把对应行从 `🔴 Open` 段移到 `🟢 Resolved` / `⚫ Wontfix` 段
- INDEX.md 不写报告内容；只列日期 + 主题 + 链接

### 4.3 报告内容结构

```markdown
# [诊断标题]

- **诊断时间**：YYYY-MM-DD HH:mm
- **状态**：open / resolved / wontfix
- **关联需求讨论**：`requirement-discussions/xxx.md`（无则写"无"）
- **触发红线**：[R-X 列表]
- **无关红线已检查**：[逐条 ID]
- **严重等级汇总**：P0: X 个 / P1: X 个 / P2: X 个 / P3: X 个

## 阶段 1：测试执行记录
## 阶段 2：发现问题
## 阶段 3：原因分析
## 阶段 4：修复方案草案
## 阶段 5：影响评估

## 违规标注（如有）
> ⚠️ 本次违规：[具体内容]
```

### 4.4 保存动作

完成阶段 5 后：
1. 汇总为 Markdown
2. 写入 `test-reports/<filename>.md`
3. 同步 `test-reports/INDEX.md`
4. 向用户确认：`✅ test-report 已保存至：test-reports/<filename>.md`

## 5. 工具使用指引

| 阶段 | 通用动作 |
|:---|:---|
| 测试执行 | 按项目实际工具栈（详见 `project-constraints.md`） |
| 截图证据 | 按项目实际截图工具（详见 `project-constraints.md`） |
| 文件写入 | Write 工具写入 `test-reports/` 目录 |
| 目录创建 | 不存在则创建 |
| 查阅参考 | 按需读取 `contracts/` / `docs/` / `requirement-discussions/` / 项目代码 |

> 项目特定的测试自动化工具（如 MCP 工具、IDE 插件、远程调试端口等）维护在 `.agents/project-constraints.md`，本文件不持有项目细节。

## 6. 允许与禁止（速查）

| ✅ 允许 | ❌ 禁止 |
|:---|:---|
| 阅读/搜索所有项目文件 | 修改任何项目代码文件 |
| 读取 `contracts/` / `docs/` / 讨论 / 报告 | 写入除 `test-reports/` 以外的目录 |
| 执行测试 / 采集日志 / 截图 | 跳过完整诊断工作流任一阶段 |
| 创建 test-reports 目录及报告 | 在未保存报告前结束完整诊断 |
| 快速诊断时直接回答不建文档 | 在报告中包含未经证实的猜测 |
| 在阶段 4 引用已有 REQ 讨论 | 忽略已有 REQ 讨论而重复设计方案 |
| 在阶段 5 引导切换到 DEV | 在用户未要求时主动修改代码 |

---

## 7. 异常处理

| 场景 | 处理方式 |
|:---|:---|
| 测试工具连接失败 | 提示用户检查项目对应工具配置（详见 `project-constraints.md`） |
| 因权限问题无法创建目录或写入文件 | 提示用户手动创建并赋予写入权限 |
| 问题根因无法在当前信息下确定 | 阶段 3 明确列出需要补充的信息，**暂停诊断等待用户提供** |
| 诊断中发现与已有 REQ 讨论冲突 | 标注冲突点，建议用户重新审视讨论或更新状态 |
| 测试环境与生产环境行为不一致 | 报告中标注 `⚠️ 环境差异`，建议真机/真环境复测 |
