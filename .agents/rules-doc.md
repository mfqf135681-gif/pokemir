# 文档模式规则 (DOC Mode)

## 1. 核心定位

**仅修改文档 .md 文件**。不修改业务代码、不修改契约、不修改 `.agents/` 本身。

**适用文件范围**：
- `docs/` 下的 `.md` 文件
- 项目根级文档（`README.md` / `CHANGELOG.md` 等）

**禁止文件范围**：
- 业务代码（任何编程语言源文件）
- 契约文件（`contracts/`）
- 治理文件（`AGENTS.md` / `.agents/*`）
- 历史归档（`change-logs/` / `test-reports/` / `requirement-discussions/` 中已存文件）

**输出风格强制约束**：遵守 `.agents/communication.md`。
**项目红线核验**：见 Router 第三步。

---

## 2. 进入前显性判断（必做）

**判定前必须先 Read 所有受影响文档**，未 Read 即输出判定视为违规。

DOC 模式开始任何 Edit 之前，回复中必须**显性输出**对**每个即将修改的文件**的判定：

#### 单文档场景

> **本次改动判定**：[仅文字层面 | 触及语义]
> **依据**：一句话说明

#### 多文档场景（必须逐文件独立判定）

> **本次改动判定（逐文件）**：
> - `docs/A.md` §X：[仅文字层面] —— [依据]
> - `docs/B.md` §Y：[触及语义] —— [依据]
> - `docs/C.md`：[仅文字层面] —— [依据]

#### 判定规则

- **任一文件判定为「触及语义」** → 整个 task **立即停止**，按 §8 异常处理建议用户切换 Mode
- **不允许"整体判定为仅文字"掩盖单文件触及语义**——多文档改动必须逐项 audit
- **若同一文件不同章节判定不同** → 拆为两条独立判定
- **边界场景**：多文档中仅 1 个文件触及语义 → 整 task 停止，建议拆分（仅文字部分可作为新 DOC task 立即推进，触及语义部分转 REQ）

> ⚠️ 这一步是 DOC 模式的核心安全阀——避免文档承载的契约/业务规则被静默改动。

---

## 3. 模式判定：快速改文档 vs 完整文档更新

| 条件 | 走哪条路径 |
|:---|:---|
| 满足以下**全部**：<br>① 改动**单个文档**<br>② 改动**单个章节**<br>③ 不涉及契约 / 数据模型 / 接口规约的语义变更<br>④ 改动行数 < 30 | **快速改文档**：直接 Edit。**Router 第四步自检仍需显性打印**，N/A 注明触发条件 |
| 任一不满足 | **完整文档更新**：执行 §5 完整工作流 |

---

## 4. 关键判断：哪些改动属于"语义变更"

**触及语义** = 必须停止并切换 Mode（不在 DOC 模式处理）：

- ❌ 修改契约文件中任何字段的类型 / 必填 / 默认值
- ❌ 修改 API 接口的入参 / 出参描述
- ❌ 修改业务流程的状态转换 / 触发条件
- ❌ 删除任何字段 / 集合 / 接口 / 流程
- ❌ 修改业务规则（如分成比例 / 合作模式 / 收费标准 / 权限矩阵）
- ❌ 修改合规要求 / 隐私条款 / 用户协议
- ❌ 修改 SLA / 性能指标 / 容量上限

**仅文字层面** = 可在 DOC 模式处理：

- ✅ 修正错别字 / 标点
- ✅ 改进描述措辞（不改语义）
- ✅ 补充示例 / 补图
- ✅ 增加章节说明（不引入新规则）
- ✅ 调整 markdown 排版（不动章节顺序）
- ✅ 修正交叉引用链接
- ✅ 补充已对齐内容的更详细解释

> ⚠️ **边界判断不确定时，倾向于"切回 REQ"**。
>
> ⚠️ 即使是"补充示例"，也要核对示例本身是否**新增了未在契约/规则中定义的行为**——若是，则属于语义变更。

---

## 5. 完整工作流

### 5.1 快速改文档

直接执行：
1. Read 目标文档
2. Edit 改动
3. 检查交叉引用受影响（参见 §6）
4. 输出 diff 摘要 + Router 第四步自检显性打印

### 5.2 完整文档更新

1. **拉清单**：列出本次要改的所有文档 + 章节
2. **依据查阅**：列出每条改动的依据来源（用户指令 / 已有 `confirmed` 讨论 / 契约文件 / 错别字等显性事实）
3. **逐文档执行**：按 Edit 工具最小改动原则修改
4. **交叉引用同步**：对照 §6 表格逐项检查
5. **diff 摘要输出**：每文件一行，格式 `<路径>: <一句话变更说明>`
6. **Router 第四步自检显性打印**

---

## 6. 文档间一致性维护

修改任一文档时，**必须检查**以下通用交叉引用（项目特定的额外引用规则维护在 `.agents/project-constraints.md`）：

| 修改类型 | 必须同步检查的内容 |
|:---|:---|
| 文档目录路由（README / INDEX） | 所有受影响的链接是否仍有效 |
| 数据模型描述 | API 契约描述中是否引用了相同字段 |
| API 接口描述 | 业务流程文档中是否引用了该接口 |
| 业务流程描述 | 数据模型中是否定义了对应状态枚举 |
| 业务对齐文档（商业模式 / 合规 / 用户协议） | 需求清单中相关条目是否仍准确 |
| 任何术语用法 | `.agents/project-glossary.md`（如存在）中的标准口语解释是否需要同步 |

**铁律**：发现文档间冲突 = 阻塞错误。立即停止并报告，不擅自决断哪一份是对的。

---

## 7. 文档风格约束

- **保持现有格式**：表格 / 列表 / 代码块的格式与周围一致
- **保持用词一致**：项目专有术语用法稳定（首查 `.agents/project-glossary.md`，无则查既有文档高频用法）
- **保持代号一致**：项目内代号风格不混用
- **章节编号**：不轻易调整章节顺序（会影响所有交叉引用）
- **图表语法**：尊重项目已有约定（Mermaid / PlantUML / ASCII 等），不切换语法
- **不"顺手"重构**：DOC 模式禁止对任务边界外的文档结构做"优化"

---

## 8. 异常处理

| 场景 | 处理方式 |
|:---|:---|
| 文档间冲突 | **停止**，列出冲突点，等用户决策哪一份为权威 |
| 改动涉及语义变更 | **停止**，告知用户："这涉及 [契约/数据模型/流程/业务规则] 变更，建议先切换到 **REQ** 讨论方案，确认后再走 **DEV** 模式实施" |
| 改动涉及 `AGENTS.md` / `.agents/` | **停止**，告知用户："这是治理变更，DOC 模式不处理。需要用户显式批准的协议修订流程" |
| 改动与已有 `confirmed` requirement-discussion 冲突 | **停止**，引用冲突的讨论记录，建议先讨论再改 |
| 改动涉及红线触发 | **停止**，按 `project-constraints.md` 对应红线合规动作处理 |
| 文档中发现明显错误事实但修复涉及语义 | **停止**，告知用户"文档与代码事实不符，建议进入 TEST 模式确认事实，再走 REQ 决定文档与代码哪一边对齐" |

---

## 9. 允许与禁止（速查）

| ✅ 允许 | ❌ 禁止 |
|:---|:---|
| 修改 `docs/` 下的 `.md` 文件 | 修改 `AGENTS.md` / `.agents/` |
| 修改 `README.md` / `CHANGELOG.md` | 修改 `contracts/` 任何文件 |
| 修复错别字 / 补示例 / 改措辞 | 修改集合字段 / 接口签名 / 状态机描述 |
| 通读上下文做必要的交叉引用同步 | 修改已归档的 `change-logs/` / `test-reports/` / `requirement-discussions/` |
| 调整 markdown 排版（不动章节顺序） | 修改业务代码 |
| 加图表（已有语法约定下） | "顺手"重构文档结构 |
| 在不确定时切回 REQ 模式 | 静默判定"语义变更"为"文字变更" |

---

## 10. 持久化策略

**DOC 模式不产出独立归档文件**。理由：
- 文档变更已由 git 历史完整追溯
- 增加 change-log 层级会让小到错别字的修改也产生 ceremony
- 与其他 Mode 的产出物（change-log / test-report / req-discussion）形成清晰分工：**代码/测试/讨论需要语义化归档，文档变更归档即 git diff**

**例外情况**：若一次性 DOC 任务涉及 ≥ 5 个文档且影响业务对齐文档（商业模式 / 合规 / 用户协议），建议在 `requirement-discussions/` 留一份对齐记录（仍按 REQ Mode 触发），而不是在 DOC 模式直接落地。

**Router 第四步自检的 DOC 强制项**：

```
□ 仅修改了 docs/ 或 README.md / CHANGELOG.md？
□ 已输出 diff 摘要（每文件一行：路径 + 一句变更说明）？
□ 进入前的「语义判定」已显性输出？
□ §6 交叉引用同步已逐条检查？
□ 触发的红线已显式引用 ID？
```
