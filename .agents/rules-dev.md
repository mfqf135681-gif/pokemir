# 开发模式规则 (DEV Mode)

## 1. 适用范围

代码编辑、新增、修改任务。

**权威来源**（只读 —— 本模式下禁止修改）：
- `docs/*` — 系统文档
- `contracts/*` — 契约文件（API 契约 + 数据模型契约，按项目实际命名与技术栈）
- `tests/*` — 测试套件
- `requirement-discussions/*` — 需求讨论记录（`confirmed` / `accepted` 状态为权威依据）

**输出风格强制约束**：遵守 `.agents/communication.md`。
**项目红线核验**：见 Router 第三步；红线详细描述在 `.agents/project-constraints.md`。

---

## 2. 实现前规则

编写任何代码之前：

- **查阅需求讨论**：在 `requirement-discussions/` 搜索相关讨论
  - 存在 `confirmed` / `accepted` 记录 → **必须**作为实现依据，「假设清单」中引用文件路径
  - 存在但状态 `pending` → 提醒用户先完成确认再动手
  - 无相关记录 → 留意是否存在需要先讨论的方案盲区
- **列前提假设**：每条标注出处（`contracts/<file>#<path>`、`docs/xxx.md §N`、`requirement-discussions/xxx.md §N` 等），无法确认来源标 `UNKNOWN`
- **不确定即标 UNKNOWN**：禁止猜测字段名 / 响应体 / 业务逻辑
- **遇歧义即停止**：多种合理解读 → 暂停请求澄清，禁止默认选择。**询问格式遵守 `communication.md §2` 三件套**（❓问什么 / 🎯选项+预测 / 💡推荐）
  - 若歧义涉及方案方向 → 建议进入 REQ 模式
- **禁止隐式假设**：每个决策可追溯至「契约 / 需求讨论 / 文档 / 用户明确指令」四者之一，均无依据则停止
- **需求不清晰 → 停止**：输出不清晰之处、所需信息、应向谁确认

### 2.0 同主题 change-log 查重（必做）

DEV 启动任何编辑之前，**一律先 `grep` 同主题关键词扫描** `change-logs/` 目录（成本低，统一做）。

| 命中情况 | 处理方式 |
|:---|:---|
| 命中同主题 change-log（同一模块 / 同一功能 / 同一 bug） | 判断任务关系：<br>• **续接型**（修补、完善同主题先前实现 / 同一 bug 的二次修复） → 新 change-log 文件名加 `_v2_` / `_补丁_` 等标识，**在 §1 任务概述显式 cross-link 前一份**<br>• **独立型**（同模块但不同功能） → 新文件，§1 末尾加「相邻任务」段引用前一份 |
| 命中但是被 REQ 讨论后否决/调整的方案 | 标注"前次方案已调整，本次按 confirmed 讨论 X 重新实现"，cross-link discussion |
| 无命中 | 按 §10 正常创建 change-log |

**关键词归一化规则**（与 `rules-req.md §2.1` 一致）：
- 剥离日期前缀 + 修饰词（"补丁" / "修复" / "再次" / "继续" / "优化" / "完善" / "重构" / "调整" 等）后的核心名词

## 3. 实现规则

编写代码时：

- **只实现当前任务**：禁止添加未被要求的功能 / 工具函数 / 抽象层 / 配置
- **禁止投机性扩展**：禁止"顺手"重构、过早泛化、"预留"参数
- **最小实现优先**：选择满足需求的最小改动
- **遵守契约**：
  - 所有请求/响应体与项目 API 契约一致
  - 所有数据写入与项目数据模型契约一致
  - 禁止发明契约未定义的字段 / 集合 / 响应属性
  - 禁止向客户端返回契约未记录的字段
- **保持现有模式**：代码风格、命名、结构与周围一致
- **遵循设计规范**：UI 改动遵循项目设计规范（如 `docs/design-guidelines.md`，按项目实际路径），无则跳过
- **响应/调用格式**：项目特定的统一壳子或调用层规范详见 `project-constraints.md`

## 4. 修改规则

修改已有代码时：

- **精确范围**：只触碰任务直接涉及的文件和代码行
- **禁止无关重构**：任务边界外不重命名 / 不重组 / 不"优化"
- **禁止外观修饰**：未发生功能变更的文件不调空白 / 不改注释 / 不重排 import
- **保留历史代码**：除非用户明确要求，不删除已有代码（即使看似未使用）
- **只清理自己产生的冗余**：仅移除当前任务引入的死代码
- **冲突即停止**：修改若影响以下任一内容，**立即停止并报告冲突**：
  - 契约文件中定义的 API 签名或响应体结构
  - 契约文件中定义的数据模型
  - 被其他模块消费的公共接口
  - `project-constraints.md` 中任一红线触发
  **例外**：`requirement-discussions/` 中 `confirmed` 记录已明确批准对应变更 → 按记录执行并在 change-log 引用

> ⚠️ **跨副本同步**：若项目存在共享逻辑物理复制点（如同一函数在多文件内联），具体清单与同步动作详见 `.agents/project-constraints.md` 对应红线。

### 4.1 5 分钟规则（顺手修复明显错误的例外）

任务边界外的**明显错误**（错别字 / 未使用 import / 格式错乱）且修复 **< 5 分钟** → 可顺手修复。但必须满足：

1. **数量 ≤ 3 处/任务**
2. **不涉**业务逻辑 / 契约 / 命名规约 / 多文件耦合
3. **change-log §3 末尾另列「附带修复」段**：每条 = 路径 + 一行说明

**判断标准**：若修复一处需要看上下游 / 跑测试验证 / 改文档同步 → **不算 5 分钟，禁止顺手做**。不确定时不修。

**超限处理**：若发现 > 3 处符合白名单的明显错误，**一律不顺手修任何一处**——避免选择性修复造成不齐。剩余项作为相邻任务建议给用户。

## 5. 验证规则

每次实现完成后：

### 5.1 判定：完整验证 vs 快速验证

| 条件 | 走哪条路径 |
|:---|:---|
| 满足以下**全部**：<br>① 改动**单个文件**<br>② 改动 **≤ 20 行**<br>③ 不涉及**接口签名变更** / **数据库 schema 变更** / **公共模块变更**<br>④ 改动类型为**纯 UI 调整**或**单行逻辑修正** | **快速验证**：手动确认改动生效。change-log 标注"轻量改动，手动验证通过"。Router 第四步自检仍需显性打印，N/A 注明触发条件 |
| 任一不满足 | **完整验证**：执行下方完整流程 |

### 5.2 完整验证流程

- **开始前定义可验证目标**（如"页面渲染正常" / "API 返回正确响应" / "测试 X 通过"）
- **执行项目测试套件**：按项目实际命令运行
  - 项目特定的测试命令、特殊环境配置（如非标准运行时路径、依赖项启动方式）详见 `project-constraints.md`
- **所有测试必须通过**。任一失败：
  1. 诊断失败原因
  2. **修复实现代码**（而非修改测试）
  3. 反复执行直至全通过
- **例外——经批准的契约变更**：若失败的唯一原因是同一任务中经用户明确批准修改了契约，或 `confirmed` 记录已确认契约修订 → 允许更新测试匹配新契约。change-log 记录"变更了哪条测试 + 原因"
- **禁止在测试未全通过时结束任务**

### 5.3 Commit + Push 默认行为（半自动机制）

**授权来源**：`requirement-discussions/2026-05-22_20-23-00_DEV结束自动commit和push.md`（confirmed）

DEV 模式任务在 §5.2 验证全通过 + change-log §1-§9 写完后，**默认自动 commit + push**（前提：6 项护栏全部通过）。

#### 护栏 checklist（任一不过 → 停手让用户介入，不自动 push）

1. ✅ **本次改动文件全部在安全列表内**：
   - 允许：`tools/*` · `tests/*` · `change-logs/*.md` · `requirement-discussions/*.md` · 业务代码（`capture/` · `events/` · `pipeline/` · `recognition/` · `storage/` · `api/`）· `docs/*.md` · `.gitignore` · `rois/*`（ROI 配置 = 项目数据，非密钥非契约）
   - **禁止自动**（除非当次 REQ 明示批准这条 .agents 改动）：`.agents/*`（治理规则）· `contracts/*`（数据/API 契约）· `.env*` / `*.key` / `*.pem` / `*.token` · `docker-compose.yml` · `requirements.txt`
2. ✅ **Secrets 扫描**：本次 `git diff` 不含 `password=` / `token=` / `Bearer ` / 长度 ≥ 32 的 hex 字串
3. ✅ **规模限制**：单文件 ≤ 5MB；本次累计改动文件 ≤ 20 个
4. ✅ **pytest 0 failed**（§5.2 全通过）
5. ✅ **change-log §1-§9 完整**
6. ✅ **working dir 干净**：除本次 DEV 改动外，**没有用户手写未 commit 的文件**（用 `git status` 核；防止误带）

#### 护栏触发时的 fallback

- 跑到 `git add` 那步停下
- 跟用户报告"护栏第 N 项命中"+ 具体证据
- 用户决定后续动作（可能：跳过那部分手动处理 / stash / 一起 commit / etc.）

#### Commit message 约定

```
<verb> <subject>

<一两句概述本次干啥>

change-log: <change-log 文件名>
Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
```

#### 明确**不在**授权范围（仍需用户确认）

- `git push --force` / `--force-with-lease`
- `git commit --amend`
- `git reset --hard` / `git revert`
- 删 branch / 删 tag
- 任何 Win 端 git 操作（我不操作 Win 机）
- TEST / REQ / DOC 模式结束时（仅 DEV 模式生效）

### 5.4 Destructive 操作护栏

**核心**:**"我能不能 X"答前必走 3 步自查** — 列全工具路径(MCP / Bash / Edit / Write / Skill)→ 逐路径评估读/写/删 → 综合结论。**禁止凭单一工具下定论**(MCP read-only ≠ 我没能力,Bash 仍可调 docker exec psql / Python)。

**Destructive 分级 + 护栏**:

| Level | 例子 | 护栏 |
|:---|:---|:---|
| L1-L2 | 本地文件修改 / 删除 | 允许,验证非 `.env` / `.docker-data/` / 红线配置 |
| L3 | git push | §5.3 半自动 6 护栏 |
| L4 | DB INSERT / UPDATE 单行 | 明确授权 + dry-run preview + change-log |
| **L5** | **DB destructive**(TRUNCATE / DELETE / DROP / ALTER 删字段) | **4 步**:(a) dry-run 列影响表+行数 (b) 用户单 turn keyword 授权(键入 yes / "清空 DB" / "TRUNCATE X") (c) 执行 + 前后行数对比 (d) change-log + 备份建议 |
| **L6** | **系统 destructive**(`docker compose down -v` / `rm -rf 数据目录`) | 同 L5 + 可逆性评估 |
| **L7** | git destructive(force-push / 删分支 / amend published) | §5.3 已挡 — 必须单 turn 明确授权 |

**授权颗粒度**:
- ✅ **强**("清空 DB" / "force push" / 键入 yes)→ 当前 L5+ OK
- ❌ **弱**("继续" / "OK" / "你看着办")→ 不构成 L5+ 授权
- ⚠️ **历史**:仅类型完全相同 + ≤ 5 turns 内
- 🚨 **用户表达"累 / 喝醉 / 焦躁" + L5+ 请求** → 强制 cooldown,推迟执行,贴 dry-run 等用户清醒再 yes

详细 case log + reasoning 留痕见 [[dev-rule-data-boundary-clarity]] 记忆。源:2026-05-28 user push back 三次后立。

## 6. 契约硬约束

| 规则 | 详细说明 |
|:---|:---|
| 禁止未声明的接口调用 | 每个对外接口调用必须在项目 API 契约中有对应定义 |
| 禁止未声明的字段 | 请求体和响应体只能包含契约 schema 中的属性 |
| 禁止未声明的数据存储 | 数据库读写只能指向项目数据模型契约中定义的结构 |
| 禁止临时数据结构 | 不创建与契约 schema 矛盾或超出其定义的内联结构 |
| Schema 变更须上报 | 任务确实需要新字段/新端点 → 停止并请求契约修订；例外：`confirmed` 记录已确认 |

## 7. 异常处理

| 场景 | 要求的操作 |
|:---|:---|
| 对需求存在不确定 | **停止**。若属方案方向性 → 建议进入 REQ |
| 契约未覆盖当前场景 | **停止**，报告缺口。建议先进入 REQ 讨论 |
| 修改后测试失败 | **修复**，反复迭代直至全通过 |
| 文档与契约冲突 | **停止**，报告差异。契约具最高权威 |
| 实现会破坏其他模块 | **停止**，报告影响范围 |
| 触发红线无对应 `confirmed` 例外 | **停止**，按红线合规动作处理或回退 |
| 在规则约束下无法完成任务 | **停止**，说明阻塞原因，**禁止放宽约束** |

**绝不为推进进度而绕过约束。暂停的任务优于损坏的系统。**

## 8. 允许与禁止（速查）

| ✅ 允许 | ❌ 禁止 |
|:---|:---|
| 修改本次任务直接涉及的文件 | 修改任务边界外的代码 |
| 创建新文件（任务要求时） | 删除未被要求删除的已有代码 |
| 读取所有项目文件 / `contracts/` / `docs/` / `requirement-discussions/` | 修改 `contracts/` 而无 `confirmed` 授权 |
| 轻量改动走快速验证 | 在测试未全通过时结束任务 |
| 执行 REQ 讨论中确认的契约修订 | 自行发明契约未定义的数据结构 |
| 在 change-log 中引用红线 ID + 讨论记录 | 猜测字段名 / 响应体 / 业务逻辑 |

## 9. 手动操作提醒

部分变更需要在代码库外执行手动步骤。完成任务后，检查 `.agents/project-constraints.md` 中各红线的「合规动作」段，列出本任务触发的手动操作。

格式：
```
⚠️ 手动操作：
1. [具体步骤]
2. [具体步骤]
```

无手动操作触发则不输出此区块。

> 项目特定的手动操作清单（如部署命令、控制台操作、依赖工具配置等）维护在 `.agents/project-constraints.md`，本文件不持有项目细节。

## 10. 修改日志（change-log）

每次完成代码修改任务必须生成 change-log。

- **存储**:`change-logs/YYYY-MM-DD_HH-mm-ss_<任务简述>.md`(续接型加 `_v2` 或 `_补丁` 后缀)
- **模板**:复制 `.agents/_templates/change-log.md` 填充
- **关联需求讨论**:涉功能 / 契约 / 红线触发 → **必须** cross-link `confirmed` / `accepted` 的 req-discussion;仅 UI 微调 / typo 可写"无(轻量改动)"。**禁止写"无"偷懒** — 应填未填视为日志不合格
- **保存确认**:向用户输出 `✅ change-log 已保存至:change-logs/<filename>.md`
