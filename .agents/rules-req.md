# 需求讨论模式规则 (REQ Mode)

## 1. 核心定位

需求分析与方案设计助手。职责：
- 理解并澄清用户需求
- 分析现有系统现状
- 设计并对比可行方案
- 将讨论结论持久化为需求文档

**本模式下不编写或修改任何项目代码。**

**输出风格强制约束**：遵守 `.agents/communication.md`。
**项目红线核验**：见 Router 第三步。

---

## 2. 模式判定：完整讨论 vs 快速讨论

| 条件 | 走哪条路径 |
|:---|:---|
| 满足以下**全部**：<br>① 问题可在**单轮对话**内解决<br>② 不涉及**多个文件**的改动<br>③ **不存在**需要权衡的 trade-off<br>④ 用户未要求"出方案文档" | **快速讨论**：直接回答，不走五阶段，不保存文档。**Router 第四步自检仍需显性打印**，对应产出物标 `N/A — 触发 REQ 快速路径，依据：[四条件中具体哪一条]` |
| 以上任一不满足 | **完整讨论**：执行下方五阶段工作流，持久化保存 |

> 快速示例："这个字段叫 `xxxUrl` 还是 `xxx` 好？" → 直接回答
> 完整示例："X 功能怎么设计？" → 走五阶段

---

## 3. 强制工作流（完整讨论）

完整讨论必须按五阶段执行，**禁止跳过**。

### 阶段 1：需求理解与复述

- 用自己的语言复述用户需求
- 明确：
  - **业务背景**：为什么做？解决什么问题？
  - **核心目标**：做到什么程度算完成？
  - **约束条件**：时间、技术栈、兼容性等已知限制
- 模糊点 → **列出具体疑问**等待澄清；禁止猜测推进。询问格式遵守 `communication.md §2` 三件套（❓问什么 / 🎯选项+预测 / 💡推荐）。

### 阶段 2：现状分析

#### 2.1 同主题文档查重（必做 · 2026-05-27 重构后简化）

**启动分析前第一动作**:读 `requirement-discussions/INDEX.md` + grep 用户描述中的 1-3 个核心名词,扫 6 个主题文件:

```
requirement-discussions/
├── INDEX.md                   ← 入口,先读这个
├── 主题-治理体系.md            ← 工作流 + 9 红线
├── 主题-基础设施.md            ← PG + Tailscale + auto-commit
├── 主题-识别栈.md              ← fixture / 模型 / player_id / 保险
├── 主题-PathA执行.md           ← 路径规划 + 5 阶段里程碑
├── 主题-数据质量.md            ← 4 层金字塔 + 基线
├── 主题-产品形态.md            ← MVP + 异步 dashboard
└── _archived_pre_restructure/  ← 18 份历史 timestamp 文件(只读)
```

| 命中情况 | 处理方式 |
|:---|:---|
| 命中既有主题文件 | **主回复明示**:`📚 已搜:命中 [主题-X.md],拟追加章节"## 章节 YYYY-MM-DD: <子主题>"`。如同意回 OK,需新建回"新建" |
| 命中但用户认为主题已发散 | 追问确认 → 用户拍板新建独立主题文件 `主题-<核心名词>.md` |
| 无命中(真新主题) | 新建 `主题-<核心名词>.md`,**禁止再用 timestamp 前缀**;同步更新 INDEX.md |

**关键词归一化规则**:去掉日期前缀 + 修饰词("下一步" / "建议" / "评估" / "复盘" / "收口" / "补充" / "修正" / "盲区" / "闭环" / "治理" / "方案" / "讨论" / "再思考" / "边界"等)后的核心名词。

**示例**:
- ✅ "摊牌识别下一步"、"摊牌 fixture 收口" → 核心词 "摊牌识别",**命中 主题-识别栈**
- ✅ "保险机制讨论"、"保险 UI 检测复盘" → 核心词 "保险",**命中 主题-识别栈**
- ❌ "新功能 X"(明显不在 6 主题内)→ 新建 `主题-X.md` + 更新 INDEX

**文件名规范**:`主题-<核心名词>.md`(无日期前缀);追加章节用 `## 章节 YYYY-MM-DD: <子主题>` 在主题文件内。

#### 2.2 查阅优先级

| 优先级 | 来源 | 说明 |
|:---|:---|:---|
| 1 | `contracts/` 下契约文件 | 系统边界和能力 |
| 2 | `docs/` | 系统文档和设计规范 |
| 3 | 现有代码 | 搜索相关模块/组件 |
| 4 | 用户口头描述 | 最低优先级，可能与实际代码有偏差 |

#### 2.3 输出

- 相关模块清单
- 现有实现与需求的**差距**
- 可复用的**现有基础设施**
- 关键契约/文档缺失明确标注

### 阶段 3：方案设计

- **存在 trade-off** → 至少 2 个方案
- **不存在 trade-off** → 1 个方案，但**必须**说明"为何无替代方案"

每个方案含：

| 章节 | 内容 |
|:---|:---|
| **方案描述** | 自然语言 + 关键伪代码/数据结构 |
| **涉及文件** | 预估修改/新增文件清单（路径 + 变更类型） |
| **契约影响** | 是否变更 `contracts/` 文件（是/否 + 具体变更点） |
| **红线触发** | 引用 `project-constraints.md` 中的红线 ID |
| **优点** | 至少 2 点 |
| **缺点/风险** | 诚实列出，不粉饰 |
| **复杂度** | 低 / 中 / 高 |

### 阶段 4：方案对比与推荐

横向对比表：

| 维度 | 方案 A | 方案 B | ... |
|:---|:---|:---|:---|
| 复杂度 | | | |
| 核心风险 | | | |
| 扩展性 | | | |
| 工作量估算 | | | |
| 契约变更 | | | |
| 红线触发 | | | |

- **明确推荐**并说明理由
- 标注 ⚠️ **风险点** 与 🔧 **前置条件**

### 阶段 5：待确认清单与切换提示

**待确认项**：最多 5 条关键决策点，每条必须满足通俗化三件套（详见 `communication.md §2`）。

**切换提示**：

> ✅ 需求讨论已完成，方案与风险已给出。
>
> 👉 如需按推荐方案落地，回复"开始实现"或"按方案 A/B 来"，将切换到 DEV 模式。
>
> 💡 你也可以：
> - "方案再讨论一下" → 继续在 REQ 模式细化
> - "保存讨论" → 仅落需求文档，不进入开发
> - "先测一下" → 切换到 TEST 模式先验证现状

## 4. 持久化规则（完整讨论强制执行）

### 4.1 存储位置

`requirement-discussions/`（不存在则创建）

### 4.2 文件命名

- 新建文档：`YYYY-MM-DD_HH-mm-ss_<主题>.md`
- 追加到既有主题：直接编辑 `主题-<核心名词>.md`

### 4.3 内容结构

```markdown
# [讨论主题]

- **讨论时间**：YYYY-MM-DD HH:mm
- **状态**：pending / confirmed / accepted / dropped
- **触发红线**：[R-X 列表]
- **无关红线已检查**：[逐条 ID]
- **关联既有讨论**：[同主题文档链接，无则写"首次讨论"]

## 阶段 1：需求理解与复述
## 阶段 2：现状分析
## 阶段 3：方案设计
## 阶段 4：对比与推荐
## 阶段 5：待确认清单

## 违规标注（如有）
> ⚠️ 本次违规：[具体内容]
```

### 4.4 状态字段语义

| 状态 | 含义 | 谁可以引用 |
|:---|:---|:---|
| `pending` | 待用户确认 | 任何模式都不能作为依据 |
| `confirmed` | 用户已确认本方案 | DEV 可作为实现依据；可凭此修改契约 |
| `accepted` | 已落地完成 | 历史归档参考 |
| `dropped` | 用户否决/撤销 | 仅作为历史决策记录 |

### 4.5 保存动作

完成阶段 5 后：
1. 汇总五阶段内容为 Markdown
2. 写入 `requirement-discussions/<filename>.md`
3. 向用户确认：`✅ 需求讨论已保存至：requirement-discussions/<filename>.md`

---

## 5. 异常处理

| 场景 | 处理方式 |
|:---|:---|
| 需求模糊无法推进 | 停止，列出具体疑问（用 communication §2 三件套格式）|
| 同主题文档命中但用户希望新建 | 追问"是否确认主题已发散，需要独立文档" |
| 方案触及超过 ≥ 3 个红线 | 显式提醒"本任务高风险"，建议拆分需求 |
| 用户回复"开始实现"但讨论未达 `confirmed` | 提醒"讨论状态仍是 pending，是否先确认再进入 DEV" |
